Alfred Triage — Glossary

The nouns of the email-tagging subsystem. Adhere to these terms in every lesson and ADR.

Triage unit Category (taxonomy) Classifier Sender context Observations Sender prior Sender relationship Thread state Conflict detection Override floor Todo rubric Label reconciliation Force / backfill
Triage unit packages/db/src/schema/triage.ts:25–74
The email_triage row. Thread-keyed: primary key is (user_id, source_thread_id) — one tag per Gmail thread, not per message. Carries category, confidence, rationale, model, appliedLabelId, and source (auto vs user override). A reply re-evaluates the thread, but only when the new message is strictly newer (recency guard).
Category (the taxonomy) packages/contracts/src/triage.ts:12–44
The closed set of ten tags TRIAGE_CATEGORIES: urgentaction_neededfollow_upawaiting_replymeetingfyidonepaymentnewslettermarketing. newsletter + marketing are rail-suppressed (never get a chip or a todo).
Classifier packages/api/src/modules/triage/classify.ts:737–801
classifyEmail() — the function that turns one email + its context into a {category, confidence, rationale, todoSuggestion}. Runs a cheap model always (Gemini Flash Lite, ADR-0051), temperature 0, ~400 max tokens, wrapped in deterministic guards (conflict re-pass + override floor). It is an LLM call sandwiched between hand-written nets — not a pure prompt.
Sender context packages/api/src/modules/triage/sender-context.ts:47–95 (ADR-0042)
extractSenderContext() — pure, ~free parsing of the From: header + body. Yields fromKind (person/service/unknown), effectiveAuthor, and a botSlug for ~10 known bots (CodeRabbit, Dependabot…). Answers "who/what sent this" before any model runs.
Observations packages/api/src/modules/triage/observations.ts:137–157
The deterministic context bundle handed to the model in the prompt. Fields: senderPrior, persona, thread, knownContact, senderRelationship, gmail signals, and cheap regex content flags. This is the slot where new context plugs in — assembled by gatherObservations() in the workflow.
Sender prior packages/api/src/modules/triage/sender-priors.ts
A per-sender histogram of past categories for bulk/service senders, learned write-back over time (incrementSenderPrior()). "This address has been marketing 12 of the last 13 times." Drives the over-classification net.
Sender relationship observations.ts:149 (ADR-0059)
resolveSenderRelationship() — for human senders, a rendered prose descriptor: significance + reciprocity + org + role, or "no prior contact." Graph-derived (built from interaction history), not from the cold-start user-standing brief. This is the closest thing today to "who is this person to me" — and the focal point of the mission's gap.
Thread state packages/api/src/modules/triage/thread-state.ts
getThreadState() — sent-mail-aware view of the thread: last reply date, newest message direction (inbound/outbound), message count, recent excerpts. Lets the model tell "they're waiting on me" from "I'm waiting on them."
Conflict detection classify.ts:463–498
detectConflict() — two tightly-gated nets that trigger a second cheap pass: under-classification (security keyword present but model went passive) and over-classification (model spiked to urgent but sender is 80%+ bulk and no Gmail IMPORTANT). Cheap-model-twice instead of one expensive model.
Override floor classify.ts:505–528
applyOverrideFloor() — a deterministic backstop that forces urgent on one narrow regex: an exposed/leaked secret/key/token. The only place hand-written code overrides the model's category outright.
Todo rubric classify.ts:35–46 (ADR-0050)
Five ordered tests the model must pass to emit a todoSuggestion: obligation → significance → memorability → actionability → not-already-handled. The chosen outcome is logged as todoDecision. Gates the rail-todo proposal downstream.
Label reconciliation packages/api/src/modules/triage/tags.ts:52–97
reconcileThreadLabel() — writes the result back to Gmail: adds the one chosen Alfred label to the canonical message and strips every Alfred label from sibling messages so the thread collapses to a single tag. Runs under a per-thread advisory lock.
Force / backfill workflow-input.ts:22; email-triage.ts:202–244
input.force — bypasses the "already-tagged" skip guard so an inbox can be re-triaged after a rubric change. Without it, a thread with an applied label and no newer message is skipped.
The one asymmetry to remember: every field above is either sender-side (who they are, what they sent) or thread-side (the conversation's shape). None encode the user's absolute standing from cold-start. That blind spot is the mission.
