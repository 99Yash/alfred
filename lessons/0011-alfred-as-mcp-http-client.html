Connect outward.
No server required.

A host can consume an MCP server exactly as a browser consumes an API: by containing a client and initiating requests. It only becomes an MCP server if it exposes its own capabilities to other clients.

The one sentence to keep: Alfred needs an MCP client library and connection manager, not an Alfred MCP endpoint, to consume remote MCP servers.

The four roles are independent

RoleIn this case
AI hostAlfred: model loop, registry, policy, approvals, durable runs
MCP clientA component inside Alfred, one logical client connection per MCP server
MCP serverA remote Calendar/Notion/etc. service exposing /mcp
Downstream APIGoogle Calendar, Notion, or another system behind that server

Alfred-as-MCP-server would be a fifth, separate product direction: another host such as Claude Desktop could connect to Alfred and call Alfred-owned tools. Consuming does not imply exposing.

Streamable HTTP is HTTP, not REST semantics

REST style
GET  /tools
POST /tools/calendar.create

MCP Streamable HTTP
POST /mcp  { "method": "initialize", ... }
POST /mcp  { "method": "notifications/initialized", ... }
POST /mcp  { "method": "tools/list", ... }
POST /mcp  { "method": "tools/call", ... }

HTTP supplies transport, TLS, headers, OAuth bearer tokens, and response streaming. JSON-RPC inside the body supplies operation names, request ids, results, and errors. One MCP endpoint handles the protocol methods.

The hosted Alfred handshake

Before

Authorize the connection

For a protected remote server, Alfred completes OAuth and obtains a token whose audience is that MCP server.

OAuth callbacks are ordinary Alfred web routes. They do not make Alfred an MCP server.

1

Alfred initiates initialize

POST https://calendar.example/mcp with a JSON-RPC initialize request, Accept: application/json, text/event-stream, and usually Authorization: Bearer ....

2

The server negotiates

It returns the agreed protocol version, its capabilities, and server identity. It may return an MCP-Session-Id HTTP header.

If present, Alfred retains and sends that session id on subsequent requests.

3

Alfred marks the session ready

It sends notifications/initialized in another POST /mcp. As a notification, it has no request id; an HTTP server accepts it with 202 and an empty body.

4

Normal operation

Alfred sends tools/list, then later tools/call, always including the negotiated MCP-Protocol-Version and any session id.

Optional

Listen for server messages

Alfred may issue GET /mcp accepting text/event-stream. The server can send notifications or requests over that SSE stream.

Alfred still initiated the GET. No public Alfred MCP callback endpoint is required.

What code Alfred would actually add

Alfred process
  MCP connection manager
    ├─ load configured remote server URL + credential reference
    ├─ instantiate MCP client/session
    ├─ initialize and negotiate
    ├─ tools/list → import namespaced catalog
    ├─ reconnect / refresh schemas on change
    └─ approved dispatch → tools/call

No Alfred /mcp route required.

In production, use the official TypeScript SDK rather than writing the transport manually. The raw lab exists only to reveal what the SDK owns.

Run the HTTP handshake

node learning-labs/mcp-from-scratch/http-demo.mjs

The demo starts a real localhost HTTP server, performs initialize → initialized → tools/list, prints every header and body, then points out that Alfred initiated every request and exposed no endpoint.

Role check

A remote MCP server sends notifications on an SSE response to a GET that Alfred opened. Has Alfred become an MCP server?
