Alfred harness · Case 1

Direct chat.
No tools.

The user says something the model can answer immediately. We follow one run from the HTTP request to a completed Postgres row.

The one sentence to keep: the workflow engine owns the durable loop; AlfredAgent performs exactly one model request inside one workflow step.

Director Workflow + executor choose what runs next.
Actor AlfredAgent performs one provider request.
Film Postgres stores state and transcript between frames.

The film

Frame 01

The route records the turn

The chat route writes the user message, mints the future assistant-message id, and calls createRun for __chat-turn__.

Run: status=pending, currentStep=chat-turn, initial state + transcript persisted. The durable assistant row is written at finalization.

Frame 02

BullMQ rings the bell

The route enqueues the run id. The queue contains a pointer, not the authoritative workflow state.

Redis/BullMQ says “work on run X”; Postgres still says what run X means.

Frame 03

The executor claims one frame

runOnce leases the run, resolves the workflow and chat-turn step, inserts an agent_steps row, and starts a heartbeat.

Run: pending → running. Step identity: (runId, chat-turn, attempt).

Frame 04

The chat step builds one request

It hydrates history, freezes connected/tool/artifact context into run state, builds the system prompt, and checks its SHA-256 against state.systemPromptHash.

A timestamp does not enter the stable system prompt; fresh runtime time is placed in an ephemeral transcript suffix.

Frame 05

Prompt-cache breakpoints are attached

AlfredAgent sorts tool definitions, strips their execute functions, and marks the system block, final tool, and last transcript message with a 1-hour Anthropic cache breakpoint.

Cache order is tools → system → messages. A matching byte prefix can be read cheaply; the new suffix is processed normally.

Frame 06

The actor gets one take

agent.streamTurn() calls AI SDK streamText with stopWhen: isStepCount(1) and execute-less tools. Text deltas stream to the UI while the durable step remains open.

Important: neither the AI SDK nor the model owns a multi-turn tool loop here.

Frame 07

The model answers directly

No tool calls are returned. The step appends the provider response to the canonical transcript, persists the assistant message, and returns { kind: "done" }.

There is no visit to dispatch-tools in this case.

Frame 08

The executor commits the ending

commitStepSuccess atomically marks the step completed and the run completed, storing the final state, transcript, and output.

Run: running → completed. The worker snapshots scratch state best-effort and does not re-enqueue.

What the prompt cache is — and is not

workflow iteration
  └─ builds provider request
       ├─ stable prefix: tools + system + old transcript  ← cache candidate
       └─ fresh suffix: runtime context + newest content ← processed now

prompt cache ≠ workflow state
prompt cache ≠ model response cache
prompt cache = reuse of provider-side prompt-prefix processing

If this first request is cold, it writes eligible prefixes. A later request with the same prefix can read them. Correctness does not depend on a hit: a miss costs more latency/tokens but produces the same logical workflow behavior.
