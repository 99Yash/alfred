Alfred harness · Foundation 2

Import the schema.
Keep the gate.

MCP can remove hand-authored integration plumbing. It cannot make model output, approval edits, server claims, or external API responses trustworthy.

The one sentence to keep: MCP can change where Alfred obtains a tool schema; it does not remove Alfred’s obligation to enforce that schema.

“Schema enforcement” is three different jobs

JobWhy it existsWith MCP
Model guidanceTell the model the operation’s name, fields, types, and descriptions.Import the server’s inputSchema; do not hand-author a duplicate.
Host validationStop malformed model output or an invalid approval edit before it is persisted or dispatched.Compile/use the imported JSON Schema and keep validating.
Server validationProtect the actual capability even if a buggy or malicious client bypasses its own checks.The MCP server must validate independently and return a useful tool error.
 
WITHOUT MCP
Alfred author writes Zod schema ──┬─ show derived JSON Schema to model
                                  ├─ validate in dispatcher
                                  └─ pass typed input to native adapter

WITH MCP
MCP server publishes JSON Schema ─┬─ show it to model
                                  ├─ validate in Alfred
                                  └─ send validated input via tools/call
                                      └─ server validates again

So the likely deletion is duplicate schema authorship for imported tools, not validation. Alfred’s built-in product tools may still use Zod because their contracts, approval UI, TypeScript types, normalization, and execution all live here.

What MCP does not repair about a bad schema

  • A server may publish vague descriptions or an overly permissive schema.
  • JSON Schema can say “string”; it cannot necessarily say “this event belongs to the intended calendar.”
  • A syntactically valid call can still encode the wrong user intent.
  • Tool annotations and returned content are untrusted unless the server itself is trusted.
  • An output schema helps parsing, but Alfred still has to validate, bound, sanitize, and retain provenance.

Calendar today: one concrete chat call

Boot

Register native tools

registerBuiltinTools() inserts calendar.list_events and calendar.create_event into Alfred’s in-process registry.

Source: packages/api/src/modules/tools/index.ts.

Expose

Build the model-visible surface

The run’s exact activeTools are projected into AI SDK tools containing a description and Zod inputSchema, but no execute function.

The model sees how to call; it cannot cause the SDK to execute directly.

Propose

The model emits a call

For example: calendar.create_event({ summary, start, end, ... }). The chat step records it in pendingToolCalls and hands off to the durable dispatch-tools step.

Still only a proposal.

Enforce

The dispatcher checks every boundary

It verifies the declared name, registry entry, workflow allowlist, exact active surface, argument keys, and Zod schema. It then resolves the user’s action policy and stores an idempotent action_stagings row.

create_event is medium risk. It gates when user policy is gated; unlike high-risk tools, it may run autonomously when policy allows.

Approve

A gated call parks

Alfred interrupts the durable run. On approval it reloads the stored or user-edited input and validates the schema again before execution.

This protects against an invalid edit and against swapping the approved payload during resume.

Execute

The native adapter calls Google

It resolves a connected Google credential, checks Calendar scopes, refreshes the access token, then calls Google Calendar v3 directly.

List: GET /calendar/v3/calendars/{id}/events. Create: POST /calendar/v3/calendars/{id}/events.

Return

Validate evidence, then continue

The Google response is parsed with Zod, normalized into a compact event, recorded by the dispatcher, and appended as a tool result. The next chat-turn lets the model answer from observed evidence.

External JSON is treated as unknown even though Alfred authored the outgoing request.

If Calendar moved behind MCP

Current native pieceMCP version
Calendar tool names/descriptions/Zod schemas authored in AlfredImported from tools/list; Alfred retains a validated, namespaced catalog entry
Alfred’s executeCreateEvent adapterMCP client sends tools/call to the owning server
Alfred owns Google token refresh and REST detailsMCP server owns its downstream Google connection
Alfred’s active-tool check, workflow cap, approval, audit, idempotencyUnchanged
Alfred validates proposed and edited inputsUnchanged in purpose; validator is driven by imported JSON Schema
Alfred sanitizes and checkpoints resultsUnchanged; MCP output is another external boundary

Architectural catch in Alfred: the current registry assumes closed built-in IntegrationSlug, ToolName, and Zod-backed tools. MCP support must open that registry to namespaced external identities and accept validated JSON-Schema-backed definitions without weakening the stricter built-in path.

“Calendar events” can mean two things

  • Calendar event records: these are currently listed or created through the chat tool path above.
  • Calendar change notifications/webhooks: those would notify Alfred that Google changed independently. The repo describes Calendar events.watch as an architectural direction, but the searched runtime currently has no implemented Calendar watch/webhook path.
