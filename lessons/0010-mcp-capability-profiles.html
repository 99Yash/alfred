MCP serves several host shapes. Capability negotiation lets each host declare only the pieces its product can safely and usefully support.

The one sentence to keep: understand an MCP feature by asking which participant offers it and which host shape has the real-world object it represents.

Name the distinction precisely

Host shapeWhat it naturally ownsExamples
Local-workspace hostUser-selected directories, local processes, an always-present interactive UI, and often a developer model subscriptionIDE, coding agent, desktop assistant
Hosted agent productRemote users, OAuth connections, durable jobs, databases, queues, audit, approval UI, and server-side model billingAlfred, workflow agent, SaaS assistant

Both are production applications. “Local workspace versus hosted agent” predicts capability fit better than “desktop versus production.”

Why roots belongs to the first profile

MCP roots are client-exposed file:// filesystem boundaries. A server asks roots/list so it knows which local project directories the client says are in scope.

IDE owns /Users/yash/project
  ↓ exposes
root: file:///Users/yash/project
  ↓ server uses as workspace context
filesystem / code-analysis MCP server

Hosted Alfred does not possess the user’s laptop filesystem, so advertising roots would be fiction. Alfred should omit roots. A future server-side code workspace could expose its own sandbox directory, but that would be a new concrete use case—not a box to tick for MCP compliance.

Read initialization as a product declaration

An Alfred MCP client does not need to send the large example capability object from the documentation. A conservative first connection could declare no optional client features:

{
  "method": "initialize",
  "params": {
    "protocolVersion": "2025-11-25",
    "capabilities": {},
    "clientInfo": { "name": "alfred", "version": "..." }
  }
}

The remote server may reply with tools. Alfred then uses those tools. It must reject a later roots/list, sampling/createMessage, or elicitation/create request because it never declared those capabilities.

Design rule for Alfred: start with remote tools over Streamable HTTP, OAuth, timeouts/cancellation, catalog refresh, and Alfred’s existing dispatch gates. Add another MCP capability only when a concrete Alfred workflow requires it.
