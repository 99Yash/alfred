Alfred harness · Foundation

Let the agent act.
Now contain it.

MCP makes more sense when we invent it reluctantly. Start with a text-only model, let it reach toward your calendar, and add machinery only when the previous design fails.

The one sentence to keep: MCP is a standard conversation between an AI application and programs that offer it data and actions; the AI application still owns judgment, permission, execution policy, and durable orchestration.

Scene 0 — the model cannot touch anything

You ask: “Move tomorrow’s dentist appointment to Friday.” The model can produce excellent text, but text does not mutate a calendar. It has no calendar connection, credential, or operating-system hand.

you → model → “Sure, I moved it.”
                  ↑
          possibly fluent fiction

First requirement: some ordinary software must stand between model output and the external world.

Scene 1 — give the model a callable function

The application—Alfred—offers a typed operation:

calendar.move_event({ eventId, newStart })

The model proposes the name and arguments. Alfred intercepts that proposal, validates it, calls calendar software, and returns the observed result to the model. The model never “reaches the calendar” directly.

This is tool calling. It already works without MCP. A hard-coded TypeScript function is enough.

Scene 2 — repeat that for every application

Now Alfred needs Gmail, GitHub, Notion, Linear, local files, and databases. Claude Desktop and VS Code want them too. Without a shared boundary, every AI application invents its own answers to the same questions:

  • How do I connect and learn what operations exist?
  • How are arguments and results shaped?
  • How do I read data that is not an action?
  • How do I detect changed capabilities, failures, progress, or disconnection?
  • How does a remote server authenticate me?

This integration seam is what MCP standardizes. It uses JSON-RPC messages and a connection lifecycle so a compliant host can speak to many compliant servers without inventing a bespoke wire protocol for each pair.

Host The AI application. Alfred owns the model loop, UI, policy, and context.
Client One connection manager inside the host for one MCP server.
Server A local or remote program that advertises capabilities and handles requests.

“Server” describes a protocol role, not a place. It may be a local child process over stdio or a remote service over Streamable HTTP.

Scene 3 — the smallest useful MCP film

Frame 01

Can we understand each other?

The client sends initialize. Both sides agree on a protocol version and declare optional capabilities.

Mitigates version guessing and unsupported-feature calls. It does not establish user intent.

Frame 02

What can you do?

Alfred asks tools/list. The server returns names, descriptions, and JSON Schemas.

Mitigates hard-coded catalogs and malformed arguments. Descriptions and annotations are still untrusted claims.

Frame 03

The model proposes

Given selected schemas, the model emits a tool call such as calendar.move_event(...).

This is a proposal in model output—not execution and not authorization.

Frame 04

The host decides

Alfred checks the exact active tool, workflow scope, connected identity, argument schema, risk policy, and any required human approval.

This is Alfred’s control plane. MCP does not supply this decision.

Frame 05

The client calls

If allowed, Alfred sends tools/call. The server validates again, enforces its own access controls, invokes the calendar API, and returns content or an error.

Both boundaries validate because neither should blindly trust the other.

Frame 06

Evidence returns

Alfred records the result and gives a bounded, sanitized representation to the model, which can now answer truthfully.

The result is external input. It may be huge, malformed, stale, or contain prompt injection.

Why MCP has three server primitives

NeedPrimitiveExample
Cause work or compute somethingToolMove an event; query a database
Read addressable contextResourceA file, schema, record, or live document
Offer a reusable interaction recipePromptA code-review or incident-analysis template

Later features go in the reverse direction too: a server can ask the host to request user input (elicitation) or obtain a model completion through the host (sampling). These are protocol abilities, not permission to silently do either.

The uncomfortable truth: interoperability increases the supply of executable capability. It does not automatically increase trust. A good MCP host is therefore a security and orchestration boundary, not merely a JSON-RPC adapter.

What MCP is versus what it does

MCP isA versioned protocol, schemas, lifecycle, transports, and SDK ecosystem for exchanging capabilities and context.
MCP doesLet a host discover and invoke tools, read resources, obtain prompts, and coordinate optional features using a shared message shape.
MCP is notAn agent, model, tool-selection algorithm, permission policy, sandbox, durable workflow engine, prompt cache, integration marketplace, or guarantee that a server is honest.

Alfred’s ADR-0018 plans the client side: connect per-user servers, import their tools into Alfred’s registry, namespace them, and route calls through the same metering and approval machinery as native tools. The backend import path is still deferred. Alfred becoming a server—letting other AI hosts call Alfred’s own capabilities—is a separate direction and is also deferred.

This also reveals where the earlier lessons fit: prompt caching and deferred schemas are optimizations for how Alfred presents a large discovered tool catalog to a model. They are downstream of MCP’s main job. You can understand and implement MCP correctly without prompt caching; you cannot safely operate an MCP tool without a trustworthy host boundary.

Boundary check
