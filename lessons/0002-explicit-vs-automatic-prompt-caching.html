One practical model for automatic caching, explicit breakpoints, and what Alfred’s lazy tool surface does to both.

The one sentence to keep: cache_control marks the end of a reusable input prefix; it does not cache only the marked block, and it never caches the model’s future answer.

First, correct the mental model

A Claude request is serialized in this order:

tools → system → messages → model generates a new answer

The provider can reuse work for an identical prefix of that input. The answer being generated is output, so it is not part of this request’s cache. On the next request, however, that old assistant answer is usually sent back inside messages. At that point it is input and can be part of a cached prefix.

User 1“Check my calendar”cache_control
OutputClaude’s new answernot input yet

That marker on User 1 writes one entry for Tools A–D + System + User 1. It does not write three independent entries, and it does not cache just User 1.

Automatic versus explicit

ModeWhere you put itWhat Anthropic does
AutomaticOne top-level cache_controlPlaces a breakpoint on the last eligible input block for each request. As history grows, that endpoint moves forward.
Explicitcache_control on chosen tool/system/message blocksCreates/checks prefix entries at exactly those endpoints, giving control over stable islands and the 20-block lookback window.

Both modes use the same prefix cache. “Automatic” automates breakpoint placement; it does not use a different cache and does not make changing bytes reusable.

Alfred’s actual explicit breakpoints

AlfredAgent currently marks up to four endpoints:

  1. The alphabetically last active tool, covering the entire active tool array.
  2. The system block, covering tools plus system.
  3. Usually the last transcript message, covering tools plus system plus history.
  4. Sometimes a pre-tool-burst message, so a large result burst cannot push the prior entry outside Anthropic’s 20-block search window.

Sorting is load-bearing: the same tool set must serialize in the same order. The system prompt is pinned and excludes changing time. Fresh time and artifact references ride in a later message suffix.

Lazy tools: simulate the consequence

Choose what happens between two otherwise-identical requests.

Alfred consequence today: Alfred's client-side lazy load changes the earliest section, tools. Because the cache hierarchy is tools → system → messages, that invalidates every later prefix for the first request with the expanded surface. No breakpoint placement can rescue that non-identical prefix. If the expanded surface stays stable, the following request can hit the newly written prefix.

Why Alfred’s explicit policy is reasonable

Automatic caching would be convenient for ordinary append-only chat. Alfred has two extra needs: a stable early breakpoint before changing runtime context, and an extra boundary around large tool-result bursts. Explicit markers make those endpoints deliberate. Lazy loading still causes a cold transition, but it also keeps the steady-state tool prefix much smaller than sending every integration schema on every turn.

Provider-native exception: Anthropic's tool-search API supports defer_loading. Deferred definitions are excluded from the early tools prefix and discovered schemas are expanded later as tool_reference blocks, preserving the cache. Alfred does not currently use that representation. Continue to Case 3: deferred tools preserve the prefix.

Use the real accounting

Do not infer hits from latency. Inspect cache_creation_input_tokens and cache_read_input_tokens. Alfred already contains a two-call probe at probe-transcript-cache.ts: first call should create; second should read the old prefix and create only the appended delta.
