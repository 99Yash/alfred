Your objection is valid. Changing the ordinary tools array is expensive, but Anthropic has a special representation for lazy tools that does not change the cached prefix.

The one sentence to keep: ordinary lazy loading mutates the tools prefix; Anthropic defer_loading keeps tools out of that prefix and inserts discovered schemas later in message history.

Alfred today: client-side lazy loading

R1: [kernel tools → system → history] ◀ warm
load calendar.create
R2: [kernel + calendar.create → system → history] ◀ cold write
R3: [kernel + calendar.create → system → history] ◀ warm

The cold request is unavoidable given this representation: Alfred rebuilds the actual active tools array after system.load_tool.

Anthropic-native deferred loading

You still send every definition to the API, but mark most with defer_loading: true. Anthropic excludes those definitions from Claude’s initial rendered tools prefix. Its server-side tool-search tool returns a tool_reference, and the matching schema is expanded inline later in the conversation.

R1: [kernel → system → history] ◀ warm
discover calendar.create
R2: [kernel → system → history] + tool_reference(calendar.create)
└──────── same cached prefix ────────┘

The schema still reaches Claude, but it arrives after the stable prefix. Therefore discovery does not invalidate that prefix.

What this means for Alfred

Alfred’s current behavior is not fundamentally required. Its installed @ai-sdk/anthropic version already exposes providerOptions.anthropic.deferLoading and Anthropic’s BM25/regex tool-search tools. A migration is technically plausible.

But it is an architectural change, not a flag flip. Alfred would need to reconcile provider-specific search results with its one-model-request workflow step, provider fallback behavior, tool-surface telemetry, availability filtering, deterministic preload, and durable transcripts. The likely good design is hybrid: keep a tiny provider-neutral kernel; use native deferred search on Anthropic; retain Alfred’s current discovery path as the portable fallback.
