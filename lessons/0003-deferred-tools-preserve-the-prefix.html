Your objection is valid. Changing the ordinary tools array is expensive, but Anthropic has a special representation for lazy tools that does not change the cached prefix.

The one sentence to keep: ordinary lazy loading mutates the tools prefix; Anthropic defer_loading keeps tools out of that prefix and inserts discovered schemas later in message history.

R3: [kernel + calendar.create → system → history] ◀ warm

The cold request is unavoidable given this representation: Alfred rebuilds the actual active tools array after system.load_tool.

Anthropic-native deferred loading

You still send every definition to the API, but mark most with defer_loading: true. Anthropic excludes those definitions from Claude’s initial rendered tools prefix. Its server-side tool-search tool returns a tool_reference, and the matching schema is expanded inline later in the conversation.

R1: [kernel → system → history] ◀ warm
discover calendar.create
R2: [kernel → system → history] + tool_reference(calendar.create)
└──────── same cached prefix ────────┘

The schema still reaches Claude, but it arrives after the stable prefix. Therefore discovery does not invalidate that prefix.

defer_loading does not mean “omit this definition from the API request.” It means “keep this definition in Anthropic’s searchable catalog, but omit it from the model’s initial rendered prompt.” The search index can inspect tool names, descriptions, argument names, and argument descriptions.

Is that catalog stored somewhere?

Internally, Anthropic must build or reuse a searchable representation. Publicly, however, there is no durable catalog resource, catalog ID, invalidation endpoint, or documented catalog TTL. The API requires Alfred to resend every complete tool definition on every request. Therefore the safe contract is: the current request’s tools array is authoritative. Any internal index cache is an implementation detail Alfred cannot depend on.

One sharp edge: if preserved history already contains tool_reference(gmail.search), Alfred must continue supplying a matching gmail.search definition even if it no longer wants that tool discovered. Otherwise Anthropic cannot expand the historical reference. Alfred should separately deny execution when authorization or availability has changed.

Yes: the HTTP payload can still be huge

“Scales to thousands” means Claude can reason over a small, relevant tool set. It does not mean Alfred sends a small request. Anthropic separates four stages that are easy to mistake for one:

StageWhat happens to 1,000 definitions?
Alfred memoryThe registry holds all 1,000.
HTTP request bodyAll 1,000 complete JSON schemas cross the wire.
Anthropic search processAll 1,000 are searched in memory for this call.
Claude context windowOnly the search tool, hot tools, and perhaps 3–5 matches enter.

1,000 tools × 2 KiB each ≈ 2 MiB request body
but perhaps 5 tools × 500 tokens ≈ 2,500 model-input tokens

The numbers are illustrative, but the asymmetry is real: network bytes and model tokens are different currencies. Deferred loading optimizes context consumption, prompt-cache stability, and selection accuracy. It does not optimize serialization, upload bandwidth, or Anthropic’s search-side work. The Messages API’s documented request limit is 32 MB, so “up to 10,000 deferred tools” is only possible when the complete request still fits that byte limit.

Anthropic says built-in tool search searches the request’s definitions in memory per call and stores nothing after the response. Unloaded definitions are not model input; definitions selected and expanded into Claude’s context count as ordinary input tokens. Tool search itself has no separate server-tool charge.

If wire size matters, the design must change

DesignAlfred sendsTrade-off
Direct deferred toolsEvery complete schema on every requestNative typed tools; large request body
Anthropic MCP connectorRemote MCP URL, credentials, and toolset policyAnthropic fetches tools; public HTTPS MCP and provider credential custody
Alfred generic gatewayOne stable search/execute schemaSmall and portable; weaker native schema constraints unless Alfred adds a second step
Two-request projectionSearch first, then only selected native schemasSmall payloads; another round trip and a changed tools prefix

So the direct answer is: nothing mysterious removes those bytes. Anthropic accepts the transport cost because the larger bottleneck is usually model context and reliable selection. Alfred should measure serialized tool bytes and request latency; if those become material, deferred loading alone is insufficient.

How does the system know what exists?

API

Alfred supplies the inventory

The top-level tools array gives Anthropic the exact candidate definitions for this request. defer_loading: true says “searchable, but initially hidden from Claude.”

MODEL

Claude sees the search capability

The non-deferred search tool and its provider instructions appear in Claude’s initial context. Claude does not initially see the hidden tool list; it turns the user’s task into a search query.

HINT

The stable system prompt can advertise categories

Anthropic recommends a compact summary such as “You can search Slack, GitHub, and calendar tools.” This improves discovery without injecting every schema into the stable prefix.

RESULT

Selected definitions become visible

Anthropic returns references and expands their matching full definitions inline. Only then does Claude learn their exact parameters and call them.

The film: discover and call one tool

Frame 01

Claude sees only the kernel

The cached prefix contains non-deferred tools, system instructions, and existing messages. Deferred schemas are absent from this model context.

Frame 02

Claude calls the search tool

For BM25 search it emits a natural-language query such as create a calendar event. This is a provider-executed server_tool_use, not an Alfred integration call.

Frame 03

Anthropic searches the hidden catalog

The server returns a tool_search_tool_result containing tool_reference(calendar.create). Up to five references are returned by default.

Frame 04

The API expands the reference

Before Claude continues, Anthropic replaces the reference conceptually with the full calendar.create definition at this point in the conversation. Alfred does not perform this expansion.

Frame 05

Claude calls the real client tool

Now that it knows the exact schema, Claude emits an ordinary tool_use(calendar.create, input). The response stops so Alfred can authorize and execute that call.

Frame 06

Alfred continues normally

Alfred executes only the real client tool and returns its tool_result. It must preserve the server-search blocks unchanged and send the same full tools array again.

Why the discovered tool stays available

The next request includes the previous assistant response unchanged: its server_tool_use, search result, tool_reference, and real tool_use. Anthropic expands references throughout history, so Claude can reuse calendar.create later without searching again.

cached old prefix + search call + tool_reference → expanded schema + real tool call + Alfred tool result ◀ new growing-history breakpoint

The early prefix stays reusable. The discovered schema and subsequent events are ordinary new suffix tokens and can themselves become part of a later growing-history cache entry.

Three jobs, three owners

AlfredSends the complete catalog, executes client tools, preserves response blocks, and enforces authorization.
Anthropic APIBuilds a searchable view from this request, runs search, expands references, and maintains the separate prompt cache.
ClaudeChooses a search query, sees the selected schema, and forms the real tool call.

What this means for Alfred

Alfred’s current behavior is not fundamentally required. Its installed @ai-sdk/anthropic version already exposes providerOptions.anthropic.deferLoading and Anthropic’s BM25/regex tool-search tools. A migration is technically plausible.

But it is an architectural change, not a flag flip. Alfred would need to reconcile provider-specific search results with its one-model-request workflow step, provider fallback behavior, tool-surface telemetry, availability filtering, deterministic preload, and durable transcripts. The likely good design is hybrid: keep a tiny provider-neutral kernel; use native deferred search on Anthropic; retain Alfred’s current discovery path as the portable fallback.
