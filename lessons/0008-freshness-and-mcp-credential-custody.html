Calling is not
observing.

A tool lets Alfred ask what the calendar looks like now. It does not make Alfred aware when the calendar changes. Authentication creates another two-hop boundary: Alfred to the MCP server, then the server to Google.

The one sentence to keep: MCP standardizes requests between Alfred and a server; it does not automatically synchronize the server’s underlying world or decide where that server keeps downstream credentials.

MCP does not alter this distinction. An MCP Calendar server can perform the live read for Alfred. To provide proactive awareness, that server or Alfred still needs to watch/poll Google, retain a cursor or snapshot, detect the relevant change, and send/consume an appropriate notification. MCP has notification and resource-subscription mechanisms, but it does not invent the Google watch or the product policy for reacting to a reschedule.

OAuth: there are two doors, not one

Door A                              Door B
Alfred ── MCP access token ──▶ MCP server ── Google access token ──▶ Google
          audience: MCP server               audience: Google APIs

The two tokens serve different resource servers and must not be treated as interchangeable.

Grant A

Alfred is authorized to call the remote MCP server

For Streamable HTTP, Alfred acts as the OAuth client. It obtains an access token specifically audience-bound to that MCP server and sends it on MCP HTTP requests.

Alfred must securely retain whatever client-side token material is needed to reconnect or refresh.

Grant B

The MCP server is authorized to call Google

If the MCP server is a Calendar proxy, it separately acts as an OAuth client of Google. Its operator stores the user’s Google grant and refreshes it.

The MCP specification explicitly forbids accepting or transiting unrelated downstream tokens as if they were MCP access tokens.

Alfred’s actual and planned storage

  • Today: native Google access and refresh tokens are columns in Postgres integration_credentials. They are currently plaintext at the application layer; Railway provides disk/volume encryption at rest. Column/application encryption is an explicit TODO.
  • Old MCP design sketch: ADR-0018 proposes mcp_servers.credentials_ref, meaning the server row points to credential material rather than embedding it alongside the catalog. No table or MCP auth flow is implemented yet.
  • Recommended implementation: keep MCP connection metadata and non-secret token metadata in Postgres, but put refresh tokens/client secrets behind an encrypted secret boundary and store only a reference. Bind each credential to user_id + canonical MCP server URI + granted scopes.

Security rule: never take Alfred’s Google token and forward it as the bearer token to an unrelated MCP endpoint. The MCP token must be issued for the MCP server; a proxy server uses a separate Google token downstream.
