Two conversations.
One host between them.

tools/list and tools/call belong to the MCP client-server wire. The model participates in a separate provider API conversation.

The one sentence to keep: MCP standardizes how the host obtains and invokes capabilities; the host translates those capabilities into whatever tool format its model provider understands.

The translation bridge

MCP conversation                    Model conversation

Alfred MCP client ──tools/list──▶ server
                  ◀──schemas─────
          │
          │ Alfred filters, namespaces, and converts schemas
          ▼
Alfred host ──selected tool definitions──▶ model
            ◀──tool proposal──────────────
          │
          │ Alfred validates + authorizes + approves
          ▼
Alfred MCP client ──tools/call───────────▶ server
                  ◀──tool result──────────

The model usually sees the server-defined tool name and schema. It does not need to know that the definition came from MCP, and it normally never emits the MCP method tools/call. That is why Alfred’s ADR says the agent should not distinguish native from MCP-imported tools at the registry layer.

Build the wire without an SDK

The lab uses the stdio transport. The client launches the server as a child process. Every line on stdin/stdout is one UTF-8 JSON-RPC message; server logs would have to use stderr.

node learning-labs/mcp-from-scratch/client.mjs

1

initialize

The client proposes protocol version 2025-11-25. The server agrees and declares the tools capability.

2

notifications/initialized

The client says initialization is complete. It has no id, so it expects no response.

3

tools/list

The universal method returns a server-specific catalog containing calendar_reschedule_preview and its JSON Schemas.

4

Model proposal

The lab prints a simulated host/model exchange. This is deliberately labeled “not MCP traffic.”

5

tools/call

After the simulated host policy seam, the client sends the server-defined name and arguments using the universal MCP method.

Why use an SDK afterward?

The raw lab makes the contract visible, but production clients and servers should use an official SDK for message schemas, lifecycle rules, pagination, cancellation, progress, tasks, transport edge cases, and version compatibility. An SDK does not change the architecture; it stops you from hand-implementing the protocol machinery shown here.
