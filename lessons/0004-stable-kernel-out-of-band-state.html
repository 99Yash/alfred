Alfred harness · Cache case 3

Keep the kernel.
Move the bulk.

A provider-agnostic design cannot own hosted KV caches, but it can own what stays stable, what is discovered, and what never enters the transcript.

The one sentence to keep: separate tool-schema churn from tool-result bulk; they need different registries and different remedies.

What a harness can and cannot own

Harness ownsCanonical prompt segments, hashes, versions, tool catalog, object handles, provider compilation, authorization, dispatch, and cache telemetry.
Provider ownsThe actual precomputed model state, cache identity rules, TTL, minimum length, billing, and whether schemas may be injected after the prefix.

A local “prompt-cache registry” can say this logical segment is stable and has hash H. It cannot make Anthropic’s cached activations reusable by Gemini.

Three external registries

Registry 1

Prompt segments

Stable instructions, current version hashes, ordering constraints, and provider-specific compilation rules.

Registry 2

Tool catalog

Canonical names, descriptions, schemas, authorization, availability, provider representations, and version hashes.

Registry 3

Context objects

Large tool results and generated datasets stored by handle with shape, provenance, permissions, lifecycle, and query operations.

The stable execution kernel

stable system prompt stable catalog.search(query) stable object.query(handle, expression) stable code.run(source, capabilities, objectHandles) stable action.invoke(name, input) ← optional generic escape hatch

Everything else stays outside the initial prompt. A provider compiler uses native deferred tools when available. The portable fallback exposes a small fixed meta-tool surface and returns only the selected signatures or handles in later messages.

“Recompute schemas on the fly” has two meanings

MeaningResult
Rebuild schemas locally, then resend them in the ordinary tools arrayNo escape: changed prefix, changed provider cache key.
Keep schemas in the registry/runtime and expose only a fixed generic executorProvider-agnostic stable prefix, but native per-tool argument guidance moves into runtime discovery and validation.

Your dataset-artifact idea

Suppose gmail.search returns 20,000 messages. Do not append the dataset to every later prompt:

tool result → object store transcript ← { handle, preview, schema, rowCount, provenance } follow-up → code.run("group/filter/join object(handle)") transcript ← small derived answer or another artifact handle

This is strong for deterministic work: filtering, aggregation, joins, statistics, sorting, and rendering. For open-ended semantic inference over every row, the model still needs selected rows, chunked model passes, embeddings, or summaries. The handle does not remove reasoning; it makes retrieval deliberate.

Why code mode is not free

  • Agent-authored code needs isolation, CPU/memory/time limits, and controlled egress.
  • Capabilities must be injected explicitly; authorization cannot live only in generated code.
  • Writes still require idempotency, approvals, audit, and durable resume.
  • Objects need lifecycle, garbage collection, provenance, and permissions.
  • The model needs discoverable types or signatures, even if it retrieves only a small subset.

Alfred’s practical first slice: park oversized tool results and return preview + handle; add a fixed read_object(handle, jsonpath, page) tool. This solves result bloat without building a code sandbox. Native provider tool search separately solves schema churn.
