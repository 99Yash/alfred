Field notes · multi-agent harness

Choosing the fleet

Every task in an agent harness needs a different amount of horsepower. Picking the right model for each is a product decision, not a user's problem — and the only way to get it right is to measure. These are the receipts behind one assistant's defaults.

01

Not every trip needs an F1 car

Models exist because they optimize for different tradeoffs — speed, cost, and reasoning depth pull against each other. The harness routes each kind of work to the cheapest model that still clears the bar.

Scan & sort
The bicycle · cheap & instant
Gemini 2.5 Flash-Lite

Classifying and triaging every inbound email, extracting facts, naming threads. High volume, low stakes.

~30× cheaper per token than the everyday model. Sub-second p50.

Do the work
The car · the daily driver
Claude Sonnet 4.6

The chat "boss," sub-agents, briefings, artifact editing. Follows instructions, uses tools, keeps the big picture.

The default for anything interactive. Reasoning-capable without overthinking.

Heavy reasoning
The cargo ship · reserved
Claude Opus 4.8

The manual "Deep" tier only. Called in for genuinely hard, open-ended reasoning — like a senior consultant you don't hand every task.

~2× the cost of the daily driver. Adaptive thinking self-sizes trivial turns to zero.

02

The candidates that didn't make it

The defaults above are survivors. Each was chosen only after a rival was tried against real, recorded tasks and set aside for a specific, measured reason. A few of the retirements:

Gemini 2.5 Flash
tested as: everyday chat model
Failed quality

Mis-orchestrated on tool-heavy turns — spun up needless sub-agents, failed to aggregate results, abandoned single-tool turns. Never cleared the bar for the driver's seat.

Gemini 3.5 Flash
tested as: everyday chat model
Quality yes, cost & reliability no

Matched or beat quality — but burned 2.6× the input and 5.3× the output tokens, landing more expensive as-run despite a lower per-token price. On multi-integration synthesis it also returned intermittent empty completions, hard-failing the turn after real spend. Kept as a cross-provider fallback only.

Claude Haiku 4.5
tested as: cheaper everyday model
Won in the lab, lost in the wild

In isolated bake-offs it hit quality parity at ~3× lower cost — briefly shipped as the default. But its one weakness (under-acting: asking instead of searching) was patched with a prompt, and the patch didn't generalize to real traffic. Reverted to the reasoning-capable model.

GPT-5.6 (two variants)
tested as: chat boss + workers
Lost on as-run cost

Competitive on per-token price, but every configuration used more turns, more tool-schema retries, and more fan-out — so the bill climbed. Registered, but not dispatched for any chat role.

A newer flagship SKU
tested as: everyday chat model
Priced out

Newer isn't automatically better-value: this one priced above the deep-reasoning tier already in use, with no default niche to justify it.

Opus 4.8 for daily editing
tested as: default artifact editor
Doesn't earn its 2×

On the dominant real workload — iterative document editing — it cost ~2× the daily driver, produced comparable design, and its whitespace-heavy layout overflowed a fixed page. Reserved for the manual Deep tier instead.

Perplexity Sonar
tested as: web-search provider
Replaced

The original web-search backend. Swapped for grounded Gemini after a billing lapse and a comparable latency profile — one fewer vendor to depend on.

03

Cheaper per token isn't cheaper

The most expensive mistake in model selection is reading the price sheet instead of the bill. What a task actually costs depends on how many turns it takes and how work is orchestrated — not the sticker rate per token.

05

Where the dollars actually go

Model selection and context management turn out to be the same problem wearing two hats. Reading the spend makes it obvious:

Cost is re-reading the conversation, not writing the answer.

Input outweighs output roughly 20 : 1. Every turn re-sends the whole transcript to the model. So a conversation that grows unchecked doesn't just get less reliable — it gets more expensive on every single turn. Controlling context growth is controlling cost.

Method. Candidates were run against real, recorded tasks through the live application, with cost recomputed from actual token usage per full trajectory — boss and every sub-agent — never from list price or benchmark scores. Defaults are re-challenged as new models ship; each figure here reflects a measured run that changed, or confirmed, a decision.

All task descriptions are generalized and contain no personal or customer data. Cost figures are the author's own testing spend. Model names are public releases.
