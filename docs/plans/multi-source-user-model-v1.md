# Multi-source user-model substrate — v1

Status: **design locked 2026-06-23** (grill-with-docs). Lands as **ADR-0067** (extends ADR-0012/0056/0057/0062; amends ADR-0057's direct-upsert capture model; under the #218 user-model spine). **Re-scopes ADR-0066/triage-user-model-v1**: that rewrite becomes a *consumer* of this graph rather than building its own entity cleanup, dist-list gate, and backfill.

> One-line: today the entity graph is a **single-source (Gmail-headers-only) pipeline of aggregate counters** with no cross-source identity, no provenance, and a scoring function that can't be recomputed. This rebuilds it as an **event-sourced, multi-source observation log** from which entities, identities, relations, significance, and facts are all replayable projections. Email becomes *one source*; calendar, GitHub, and every future integration feed the same substrate.

## Why this exists (prod findings, work account `yash.k@oliv.ai`, 2026-06-23)

A read of the prod graph surfaced the disease, not just symptoms:

- **Single-source.** Entities are mined *only* from Gmail From/To/Cc headers (`team-graph.ts`). Calendar attendees and GitHub PR people are available but **explicitly deferred** in code (`team-graph.ts:20`). The most "who matters" signal Alfred has — who you actually *meet with* and *ship code with* — never reaches the graph.
- **Misclassification.** Everything that sends mail becomes `kind: person` with a `works_at <domain>` edge. So `OlivAIRepo/baserow-middleware` is a "person" who `works_at noreply.github.com`; a dozen `dev.N@oliv.ai` test accounts are "people."
- **Distribution-list pollution.** `'Anthropic' via Engineering` (`engineering@oliv.ai`, 50 inbound) is the **#1-scored "person" on prod (0.72)** — a list alias outranking every human, silently corrupting every significance-weighted decision.
- **Flat, unrecomputable significance.** The whole standup list scores an identical `0.368`: `reciprocity` is pinned at `0.2` for inbound-only, `sameOrg=1` for everyone @oliv.ai, so only `frequency` moves — and it's driven by `coRecipient` touches (being cc'd on group threads), not real interaction. Scores live in `entities.metadata` aggregates; **change the formula and you can't recompute history.**
- **No cross-source identity.** GitHub login `sanyam-oliv` and email `sanyam.p@oliv.ai` are unrelated rows. There is no `entity_identities` concept; `aliases` is an untyped jsonb blob.
- **`user_facts` is a junk drawer.** 397 of the work account's 431 "facts" are document ephemera auto-confirmed at `0.95` — `zoom_meeting_passcode`, `cloudwatch_alarm_name`, `incident_timestamp`. Facts are user-scoped only; nothing binds a fact to the person it's about.
- **`integration_object_relations` is empty for all users** (#212 shipped objects + keys but never wired the object↔entity edges — there was no resolver to map a PR's author login to a person).

These are one disease: **the user model has the wrong substrate.** Patching the email pipeline can't fix multi-source, identity, provenance, or recompute.

## External calibration (2026-06-23)

The current market mostly validates the shape, not a vendor switch:

- **Event sourcing + materialized projections** is the canonical shape for replayable state: append events, rebuild read-optimized projections/materialized views, and use replay for audit/debug/backfill ([Microsoft event-sourcing pattern](https://learn.microsoft.com/en-us/azure/architecture/patterns/event-sourcing)). That validates the log → projection shape; it does not require changing Alfred's database.
- **Zep / Graphiti** validates the temporal-context-graph direction: episodes fold into a time-aware graph for agent memory. It also confirms why ADR-0058 still stands here: [Graphiti](https://github.com/getzep/graphiti) wants a graph backend (Neo4j/FalkorDB/Neptune) plus LLM-assisted ingestion. Borrow the temporal graph and episode/projection pattern; do not move Alfred's source of truth out of Postgres.
- **Mem0 / Letta** validate autonomous memory operations and archival semantic recall. They are not the right authoritative store for triage/briefing decisions because Alfred needs deterministic replay, provenance, reversibility, and Replicache-visible edits. Borrow operation taxonomy and memory UX; keep the store local.
- **First-party source APIs are more valuable than third-party enrichment for v1.** Gmail headers (including the [RFC 2919 `List-Id` standard](https://datatracker.ietf.org/doc/rfc2919/)), Calendar attendees plus `recurringEventId`/`originalStartTime` ([Calendar Events API](https://developers.google.com/workspace/calendar/api/v3/reference/events)), GitHub webhooks, and Google Workspace People/Directory profiles (`people.listDirectoryPeople`, which requires [`directory.readonly`](https://developers.google.com/people/api/rest/v1/people/listDirectoryPeople) and admin-enabled domain sharing per the [People API overview](https://developers.google.com/people)) are higher-trust, cheaper, and already permission-aligned with Alfred's integrations.
- **Third-party people/company enrichment is an optional enrichment lane, not foundation.** People Data Labs ([Person Enrichment API](https://docs.peopledatalabs.com/docs/reference-person-enrichment-api)), Exa People Search, NinjaPear/Enrich Layer-style APIs can help cold dossiers or missing roles later, but they add privacy/compliance/cost risk and should only write `proposed` facts with citations. Do not build a hard dependency on unofficial LinkedIn-derived vendors; [Proxycurl's 2025 shutdown](https://nubela.co/blog/goodbye-proxycurl/) is the cautionary example.

## The model in one screen

```
SOURCES (reducers, one per provider — the ADR-0062 registry pattern)
  gmail | google-calendar | github(pr/push/review) | clickup | notion | ...
        |  reduce(rawEvent) -> Observation[]
        v
observations            <-- APPEND-ONLY SYSTEM OF RECORD (event-grain)
  id, user_id, source, kind, occurred_at,
  subject_identity {kind,value}, object_identity?,
  participants jsonb (full To/Cc/attendees/reviewers + recipientCount + list_id),
  payload jsonb,
  family_key, evidence_hash, dedup_key (unique = family_key:evidence_hash)
        |
        |  PROJECTION (deterministic fold; replayable; never the source of truth itself)
        v
  +-- resolve identity ({kind,value} -> stable entity_id; deterministic + reversible merges)
  +-- entities            (STABLE node id + supersession/forwarding; no projection_version)
  +-- entity_identities   (STABLE typed identity keys, unique (user_id,kind,value), per-id provenance)
  +-- entity_profiles     (VERSIONED display/kind/significance components + read-time recency)
  +-- entity_relations    (VERSIONED typed: works_at, member_of, reports_to, frequent_collaborator, ...)
  +-- co_occurrence       (weighted pair projection; fan-out cutoff; promotion -> typed edge)
  +-- user_facts          (subject = user | entity_id; key in FACT_ONTOLOGY; ephemera excluded)
```

Everything below `observations` is derived, but with two different contracts: stable `entities`/`entity_identities` preserve foreign-key identity and merge forwarding, while versioned read models are **recomputable**. Change a weight, a cutoff, a classifier, or the ontology, and replay the fold into a new projection version without replacing stored entity IDs. Any projection row that becomes Replicache-visible has a third contract: a stable sync key plus a content version that changes only when the serialized value changes, so an active-version flip does not look like delete-all + re-add-all to clients.

## Decisions (resolved in the 2026-06-23 grill)

- **D1 — System of record = event-sourced observation log.** Append-only per-source observations; entities/identities/relations/significance/facts are replayable projections. Observations are retained indefinitely in v1: storage is the price of replay, audit, and future reducer fixes. Superseded observations stop being the active family member, but they are not pruned; only projections are compacted/rebuilt. Rejected: aggregate counters (lossy, unrecomputable), hybrid (two drifting write paths).
- **D2 — Identity = stable entity nodes + a dedicated `entity_identities` table.** Entity IDs are the stable foreign-key surface for the rest of Alfred, so they are **not projection-versioned**. `entities.id` is content-addressed from the first normalized hard identity that creates the node; later canonical-winner rules choose the merge survivor/display primary, not the already-issued ID. Merges leave a forwarding pointer (`supersedes_entity_id`) on losers and all reads resolve through `resolveActiveEntityId`. `entity_identities` stores `(user_id, entity_id, kind, value, confidence, source, verified, valid_from/until)`, unique on `(user_id, kind, value)`. Replaces the overloaded `aliases` jsonb; it is the dedup index *and* the join target observations resolve through. Kinds: `email`, `github_login`, `slack_id`, `notion_user_id`, `domain` (orgs), …
- **D3 — Cross-source merge = deterministic-first + reversible proposals.** Auto-merge only on a hard bridge (shared email; GitHub **commit-author email** from `push` payloads; a confirmed `github_username` fact; verified Google Workspace Directory profile/email when `directory.readonly` + domain sharing are available). Name/fuzzy matches are *proposed*, gated, reversible (propose/dispose, ADR-0062 invariant) — never auto-asserted. Unmapped identities live as provisional entities, merged later via supersession when a bridge arrives. **Coverage is partial by design** (masked `noreply` commit emails don't bridge to work email; fall back to the `github_username` fact).
- **D4 — Observation grain = event-grain, with evidence hashing.** One row per relationship-bearing event version (one email / meeting occurrence / PR-event evidence), full participant set in `participants`. Pairwise edges + significance derived in the fold. A 50-recipient blast is **one** observation carrying `recipientCount: 50` — the reducer down-weights it instead of minting 50 `coRecipient` touches. To keep append-only + supersession coherent, every reducer emits `family_key` (stable event identity, e.g. `gmail:<message_id>`, `calendar:<calendar_id>:<event_id>:<original_start>`, `github:pr:<repo>:<number>`) and `evidence_hash` (only relationship-significant fields: participants/attendees/reviewers/assignees + start where relevant). `dedup_key = family_key:evidence_hash`. Same evidence hash → duplicate/no-op; changed evidence → append a new observation with `supersedes_observation_id` pointing at the prior active family member. Rejected: participant-grain (re-creates today's inflation), bare occurrence-key uniqueness (blocks material changes), etag-only dedupe (trivial edits inflate history).
- **D5 — Graph shape = full social graph, made viable.** Stored as a **weighted pair projection** (`co_occurrence`: one row per distinct co-occurring pair, `weight/count/last_seen`). **Fan-out cutoff**: events above N participants contribute **zero** pairwise co-occurrence (a blast is not social evidence); below, `weight += 1/participants`. **Promotion threshold**: a pair becomes a traversable typed `frequent_collaborator` edge only past a weight bar; below that it's queryable data, never traversed. So traversal only ever walks typed + promoted edges (small, indexed). Full graph, bounded size, near-linear writes, recomputable.
- **D6 — Significance = deterministic source-weighted fold + read-time recency.** Forced deterministic by the event-sourced choice (can't replay an LLM over all history per weight change). The projection materializes only **time-invariant components** (`volume`, `reciprocity`, `sameOrg`, source-weighted interaction weights, raw co-occurrence weight, `lastSeenAt`, top contributing observations). The final score is computed at read time as `baseScore(components) * recencyMultiplier(asOf, lastSeenAt)` (e.g. exp decay), so deterministic projection checksums do not change as wall-clock time advances and no daily full re-projection is needed just for decay. One score per user (persona is already separated — work vs personal are distinct `user` rows).
- **D7 — Kind taxonomy, classified into the versioned profile.** `person | organization | group | service | repository | project`. Detection: [`List-Id`](https://datatracker.ietf.org/doc/rfc2919/)/`List-Unsubscribe` headers (authoritative) + local-part heuristics (`team@`, `all@`, `*-eng@`) + fan-out for `group`; `noreply@`/`notifications@`/`*-mirror@` for `service`; GitHub repo addresses → `repository` (linked `in_org`, not `works_at` as a person). Non-humans are **retained as typed nodes** (queryable, recomputable, no signal lost) but **never person-significance-scored**. Because better classifiers can change `kind`, it lives in versioned `entity_profiles`, not on the stable entity node. Rejected: suppress-at-ingest (loses signal), is_automated flag (conflates repo/list/bot).
- **D8 — `user_facts` = typed ontology + subject binding.** Every fact binds to a **subject** (`{kind:'user'}` or `{kind:'entity', id}`) and a `key` from a **registered durable fact-type** (`FACT_ONTOLOGY` in `@alfred/contracts`): `employer, job_title, manager, reports_to, timezone, github_username, …`. Facts about people hang off their entity node (the "personalized relevance" destination). Transient document content (passcodes, alarm names, incident times) **is not a fact-type** → stays in the document / a `memory_chunk` (RAG), never auto-confirmed as a user fact. Rejected: TTL/decay-only (leaves the junk drawer, no subject).
- **D9 — `integration_object_relations` wired through the resolver, storing stable entity IDs.** ADR-0062's object/entity separation stays (work-objects out of the canonical people table). Each GitHub webhook now emits **two** projections: observations into the social/significance graph (a review *is* a work-relationship signal) **and** the `authored_by`/`reviewed`/`assigned`/`closes` object↔entity edges via the identity resolver. `integration_object_relations.entity_id` always references the stable entity layer and reads resolve supersession forwarding, so D13 projection flips cannot dangle object links.
- **D10 — One-time substrate migration = shadow-build + validated cutover.** New tables coexist with the legacy aggregate graph. Replay all sources into the observation log; project the new graph; **validate coverage against the old graph** (row counts, expected identities present, no source family dropped) while treating legacy significance ordering as a polluted baseline that is expected to diverge. Correctness comes from deterministic replay checks and spot-checks: `engineering@`→group, repos→repository, Sanyam GitHub↔email merged where a hard bridge exists, flat-0.368 standup list differentiated. Flip consumers behind a flag, watch, then drop legacy columns/tables. Idempotent by `dedup_key`; dry-run locally; prod replay via `railway ssh`. Rejected: in-place rebuild (empty-graph window on the account you depend on), forward-only (wastes history in `documents`/`webhook_events`).
- **D11 — Sequencing = ADR + plan first, then incremental Gmail-first.** Each phase shadow-builds and validates before the next. Gmail-first because it's the source with the most data/intuition — cleanest validation of the new projection before adding sources.
- **D12 — Governance = this epic is the foundation; ADR-0066 consumes it.** No two systems on the same tables. The triage rewrite reads the new graph.
- **D13 — Steady-state projection re-versioning is checkpointed and read through active views.** Event-sourcing only pays off if replay is deterministic. Each projection records `(projection_name, projection_version, source_high_watermark, checksum, completed_at)`. Only recomputed outputs are projection-versioned: `entity_profiles` (display/kind/significance components), `entity_relations`, `co_occurrence`, and `user_facts`. Stable `entities` and `entity_identities` are not version-partitioned because external tables store `entity_id`. A weight/ontology/reducer change bumps a version, shadow-replays, compares checksums/counts/top-N at a fixed validation `asOf` when final scores are involved, then atomically flips the active projection pointer. Consumers do **not** hand-write `WHERE projection_version = active`; they read `active_entities`, `active_entity_relations`, `active_co_occurrence`, `active_user_facts`, etc. views that pin the current version and join to stable entities. Checksums are over deterministic row order (`ORDER BY id` / stable key), rounded floats, and **time-invariant components only**; final recency-decayed scores are excluded because they are read-time values. Rejected: "just rerun the script" with no replay identity; projection-versioning stable IDs (D13 flip would dangle `integration_object_relations`/facts/todos/briefings); shared versioned rows without active views (one forgotten filter gives mixed-version reads).
- **D14 — User actions are source events too, with rank-then-recency precedence.** Chat-captured standing instructions, user confirmations/rejections, manual profile edits, and correction feedback enter the same log as high-precedence `source='user'|'alfred_chat'` observations. The fold resolves conflicts by source rank first (`user` > `alfred_chat` > first-party integrations > enrichment/RAG), then recency within the same rank. Projections may propose facts from integrations, but they must never overwrite user-authoritative corrections; they add superseding observations or proposed conflicts. Existing non-proposed `user_facts` rows (`confirmed`, user-edited, rejected/dismissed, and `standing_instruction`) migrate into `source='user'` observations, then project back into `user_facts`. Status/confidence must be derivable from the fold: a confirmation/edit/rejection is itself a user observation, so replay cannot downgrade confirmed facts back to proposed.
- **D15 — Reducers distinguish raw events from relationship evidence.** A provider event can produce several observations, but only relationship-bearing occurrences affect significance/co-occurrence. Example: a Calendar event update or reminder change should not count as another meeting with the attendees; a meeting occurrence does. GitHub push/PR/review events similarly need reducer-specific evidence kinds so significance is not inflated by retries, edits, or noisy automation.
- **D16 — Stable IDs are the foreign-key contract; versions are read-model contracts.** Anything outside the projection engine (`integration_object_relations`, `user_facts.subject`, future todos/briefings/meeting-preps) stores only stable `entity_id`s. Re-versions may change kind, significance, relations, and co-occurrence, but they must not mint replacement entity IDs. Merges choose a deterministic winner, write forwarding on losers, and every read path resolves through the forwarding chain before joining active projections.
- **D17 — Replicache-visible projections need stable sync keys + content versions.** Today's sync diff is CVR over per-row `rowVersion`; a projection-version flip that swaps physical rows without stable keys would look like delete-all + re-add-all and can wipe optimistic local state. Every active view that feeds Replicache (`active_user_facts`, and any future `active_entities`/profile surface) must expose a stable logical key and a synthetic `rowVersion`/content version derived from serialized-content hashes. The projection cutover computes per-key added/changed/removed deltas: unchanged keys keep their prior sync version, changed keys bump once, removed keys delete, new keys insert. This belongs in P0 schema/read-model design and P5 cutover validation, not a later UX cleanup.

## New schema (additive; `packages/db/src/schema/`)

> All tables carry `user_id` (CASCADE per single-user FK convention), `lifecycle_dates`, and `createId` prefixes. Names indicative; finalize in P0.

- **`observations`** — `id (obs)`, `user_id`, `source`, `kind`, `occurred_at`, `family_key`, `evidence_hash`, `dedup_key`, `subject_identity jsonb {kind,value}`, `object_identity jsonb?`, `participants jsonb`, `payload jsonb`, `schema_version`, `reducer_version`, `supersedes_observation_id?`, `lifecycle_dates`. Indexes: `(user_id, source, occurred_at)`, `(user_id, family_key)`, unique `(user_id, dedup_key)`.
- **`entities` (stable)** — `id (ent)` content-addressed from `(user_id, first normalized hard identity)`, `user_id`, `canonical_identity jsonb`, `supersedes_entity_id?`, `lifecycle_dates`. No `projection_version`. External references store this ID; read helpers resolve forwarding chains.
- **`entity_identities` (stable)** — `id (eid)`, `user_id`, `entity_id` → stable `entities`, `kind`, `value`, `confidence`, `source`, `verified bool`, `valid_from/valid_until`, `supersedes_id`, source-observation provenance, `lifecycle_dates`. Unique `(user_id, kind, value)`; index `(user_id, entity_id)`. `aliases` jsonb is retired in favor of this table (kept during shadow phase, dropped at cutover).
- **`projection_runs` / `projection_cursors` / `active_projection_versions` / `projection_sync_state`** — tracks active projection versions, source high-watermarks, row-count/checksum summaries, completion status, cutover pointer, and Replicache-visible sync state `(sync_slug, stable_key, content_hash, row_version)`. This is the replay safety rail; consumers read active-version views, never unfiltered projection tables.
- **`entity_profiles` (versioned)** — `id (eprof)`, `user_id`, `projection_version`, `entity_id` → stable `entities`, `display_name`, `kind`, `significance_components jsonb`, `last_seen_at`, `provenance jsonb`, `computed_at`, `lifecycle_dates`. Unique `(user_id, projection_version, entity_id)`. Read helpers expose final recency-adjusted score from components + caller `asOf`.
- **`entity_relations` (versioned)** — `id (erel)`, `user_id`, `projection_version`, `from_entity_id`, `to_entity_id` (both stable), `relation_type`, `weight/confidence`, `provenance`, `valid_from/valid_until`, `lifecycle_dates`. Add typed relations `member_of`, `reports_to`, `frequent_collaborator`, `in_org` (repo→org); `works_at` retained for person→org. Unique/index by `(user_id, projection_version, relation_type, from_entity_id, to_entity_id)`.
- **`co_occurrence` (versioned)** — `id (cooc)`, `user_id`, `projection_version`, `a_entity_id`, `b_entity_id` (stable IDs, ordered to dedupe the pair), `weight real`, `count int`, `last_seen_at`, `lifecycle_dates`. Unique `(user_id, projection_version, a_entity_id, b_entity_id)`; index `(user_id, projection_version, weight)` for promotion sweeps.
- **`user_facts` (versioned projection)** — add `subject jsonb` (`{kind:'user'}` | `{kind:'entity', id}` where `id` is stable), `projection_version`, status/confidence, and source-observation provenance; `key` validated against `FACT_ONTOLOGY` at the app boundary. Existing non-proposed user-authoritative facts migrate into `source='user'` observations, then project back into `user_facts`.
- **Active views/readers** — `active_entities` (stable entities joined to active `entity_profiles`), `active_entity_identities` (stable), `active_entity_relations`, `active_co_occurrence`, `active_user_facts`; application code reads through these or repository helpers only. Any view wired to Replicache exposes `id = stable_key` and `rowVersion` from `projection_sync_state`, not the physical projection-row id/version.
- **`@alfred/contracts`** — `OBSERVATION_KINDS`, `SOURCE_WEIGHTS` registry, `IDENTITY_KINDS`, `ENTITY_KINDS`, `FACT_ONTOLOGY`, `FAN_OUT_CUTOFF`, `PROMOTION_THRESHOLD` — the tunable, no-migration knobs (text columns + boundary validation, same ergonomics as `INTEGRATION_OBJECT_DEFS`).

## Phases

- **P0 — Schema + contracts.** New tables, projection run/cursor tracking, Replicache projection-sync state, registries, additive migrations (`db:generate` → `db:migrate`, never push). No behavior change. P0 must lock the stable-vs-versioned split before schema generation: stable `entities`/`entity_identities`; versioned `entity_profiles`/`entity_relations`/`co_occurrence`/`user_facts`; stable sync keys + content-versioning for any active view that will feed Replicache.
- **P1 — Gmail source end-to-end + validation.** Reducer (`documents` → observations), the full projection (resolve → classify kind → fold time-invariant significance components + co_occurrence), replay-backfill from `documents`. **Validate the new graph against today's** (shadow). This proves the entire pipeline on data we understand before any new source. The biggest corruption fix lands here: `engineering@oliv.ai` / `'Anthropic' via Engineering` becomes `group` and unscored as a person, independent of the later ADR-0066 category rewrite.
- **P2 — GitHub.** `pull_request`/`push`/`pull_request_review` → observations; **commit-author-email identity bridges** (the highest-yield signal we currently discard); wire `integration_object_relations` (D9). Replay from `webhook_events`.
- **P3 — Calendar + Workspace Directory.** Calendar ingestion for event attendees/organizer (the highest-signal source missing today), using `family_key + evidence_hash` occurrence evidence so updates do not inflate scores. For recurring meetings, family identity uses the Calendar event series/instance fields (`recurringEventId` + immutable `originalStartTime` where present) from the [Calendar Events API](https://developers.google.com/workspace/calendar/api/v3/reference/events). If the Google Workspace domain permits it, People API Directory (`listDirectoryPeople`/incremental sync, scope `https://www.googleapis.com/auth/directory.readonly`) seeds high-trust identities/org membership; if not available, skip without blocking. Verify before build: add the scope to `GOOGLE_FEATURE_SCOPES` only after confirming it is grantable on the Google OAuth client and that the `oliv.ai` admin has enabled external contact/profile sharing.
- **P4 — Fact ontology.** `subject` binding + `FACT_ONTOLOGY` gate; migrate all non-proposed user-authoritative facts into user observations; reclassify/purge the `user_facts` junk drawer; attach people-facts to stable entity nodes; ensure status/confidence are replay-derived from user confirmation/edit/rejection observations.
- **P5 — Consumer cutover.** Re-point triage sender-priors / briefing / meeting-prep (and the re-scoped ADR-0066 "You" block + dist-list gate) at the new graph; flag-flip; validate Replicache patches are per-key deltas rather than wholesale replacement; watch; drop legacy `metadata.correspondence` + `aliases`.

## Backfill / rollout (D10 detail)

1. Ship P0 schema (additive, coexists).
2. Replay `documents` + `webhook_events` + a Calendar pull → `observations` (idempotent on `dedup_key`; dry-run local first). Also migrate existing non-proposed `user_facts` rows into `source='user'` observations so confirmed/edited/rejected user intent survives replay.
3. Project → stable entity layer + versioned read-model tables under a named projection version; expose versioned outputs through active-version views only after cutover.
4. **Replay twice and validate determinism:** same input high-watermarks produce identical counts/checksums and, where needed, top-N computed with a pinned validation `asOf`; projection cursors prove no observation was double-counted. Checksums use stable ordering and rounded floats over time-invariant components only; read-time recency-adjusted scores are excluded.
5. **Validate** vs old graph for coverage only: row counts, source-family coverage, and expected legacy identities accounted for. Do **not** use legacy top-N significance ordering as the correctness oracle because today's ordering is the polluted failure mode; expect meaningful ordering divergence.
