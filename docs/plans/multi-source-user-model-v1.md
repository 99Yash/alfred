# Multi-source user-model substrate — v1

Status: **design locked 2026-06-23** (grill-with-docs). Lands as **ADR-0067** (extends ADR-0012/0056/0057/0062; amends ADR-0057's direct-upsert capture model; under the #218 user-model spine). **Re-scopes ADR-0066/triage-user-model-v1**: that rewrite becomes a _consumer_ of this graph rather than building its own entity cleanup, dist-list gate, and backfill.

> One-line: today the entity graph is a **single-source (Gmail-headers-only) pipeline of aggregate counters** with no cross-source identity, no provenance, and a scoring function that can't be recomputed. This rebuilds it as an **event-sourced, multi-source observation log** from which entities, identities, relations, significance, and facts are all replayable projections. Email becomes _one source_; calendar, GitHub, and every future integration feed the same substrate.

## Why this exists (prod findings, work account `yash.k@oliv.ai`, 2026-06-23)

A read of the prod graph surfaced the disease, not just symptoms:

- **Single-source.** Entities are mined _only_ from Gmail From/To/Cc headers (`team-graph.ts`). Calendar attendees and GitHub PR people are available but **explicitly deferred** in code (`team-graph.ts:20`). The most "who matters" signal Alfred has — who you actually _meet with_ and _ship code with_ — never reaches the graph.
- **Misclassification.** Everything that sends mail becomes `kind: person` with a `works_at <domain>` edge. So `OlivAIRepo/baserow-middleware` is a "person" who `works_at noreply.github.com`; a dozen `dev.N@oliv.ai` test accounts are "people."
- **Distribution-list pollution.** `'Anthropic' via Engineering` (`engineering@oliv.ai`, 50 inbound) is the **#1-scored "person" on prod (0.72)** — a list alias outranking every human, silently corrupting every significance-weighted decision.
- **Flat, unrecomputable significance.** The whole standup list scores an identical `0.368`: `reciprocity` is pinned at `0.2` for inbound-only, `sameOrg=1` for everyone @oliv.ai, so only `frequency` moves — and it's driven by `coRecipient` touches (being cc'd on group threads), not real interaction. Scores live in `entities.metadata` aggregates; **change the formula and you can't recompute history.**
- **No cross-source identity.** GitHub login `sanyam-oliv` and email `sanyam.p@oliv.ai` are unrelated rows. There is no `entity_identities` concept; `aliases` is an untyped jsonb blob.
- **`user_facts` is a junk drawer.** 397 of the work account's 431 "facts" are document ephemera auto-confirmed at `0.95` — `zoom_meeting_passcode`, `cloudwatch_alarm_name`, `incident_timestamp`. Facts are user-scoped only; nothing binds a fact to the person it's about.
- **`integration_object_relations` is empty for all users** (#212 shipped objects + keys but never wired the object↔entity edges — there was no resolver to map a PR's author login to a person).

These are one disease: **the user model has the wrong substrate.** Patching the email pipeline can't fix multi-source, identity, provenance, or recompute.

## External calibration (2026-06-23)

The current market mostly validates the shape, not a vendor switch:

- **Event sourcing + materialized projections** is the canonical shape for replayable state: append events, rebuild read-optimized projections/materialized views, and use replay for audit/debug/backfill ([Microsoft event-sourcing pattern](https://learn.microsoft.com/en-us/azure/architecture/patterns/event-sourcing)). That validates the log → projection shape; it does not require changing Alfred's database.
- **Zep / Graphiti** validates the temporal-context-graph direction: episodes fold into a time-aware graph for agent memory. It also confirms why ADR-0058 still stands here: [Graphiti](https://github.com/getzep/graphiti) wants a graph backend (Neo4j/FalkorDB/Neptune) plus LLM-assisted ingestion. Borrow the temporal graph and episode/projection pattern; do not move Alfred's source of truth out of Postgres.
- **Mem0 / Letta** validate autonomous memory operations and archival semantic recall. They are not the right authoritative store for triage/briefing decisions because Alfred needs deterministic replay, provenance, reversibility, and Replicache-visible edits. Borrow operation taxonomy and memory UX; keep the store local.
- **First-party source APIs are more valuable than third-party enrichment for v1.** Gmail headers (including the [RFC 2919 `List-Id` standard](https://datatracker.ietf.org/doc/rfc2919/)), Calendar attendees plus `recurringEventId`/`originalStartTime` ([Calendar Events API](https://developers.google.com/workspace/calendar/api/v3/reference/events)), GitHub webhooks, and Google Workspace People/Directory profiles (`people.listDirectoryPeople`, which requires [`directory.readonly`](https://developers.google.com/people/api/rest/v1/people/listDirectoryPeople) and admin-enabled domain sharing per the [People API overview](https://developers.google.com/people)) are higher-trust, cheaper, and already permission-aligned with Alfred's integrations.
- **Third-party people/company enrichment is an optional enrichment lane, not foundation.** People Data Labs ([Person Enrichment API](https://docs.peopledatalabs.com/docs/reference-person-enrichment-api)), Exa People Search, NinjaPear/Enrich Layer-style APIs can help cold dossiers or missing roles later, but they add privacy/compliance/cost risk and should only write `proposed` facts with citations. Do not build a hard dependency on unofficial LinkedIn-derived vendors; [Proxycurl's 2025 shutdown](https://nubela.co/blog/goodbye-proxycurl/) is the cautionary example.

## The model in one screen

```
SOURCES (reducers, one per provider — the ADR-0062 registry pattern)
  gmail | google-calendar | github(pr/push/review) | clickup | notion | ...
        |  reduce(rawEvent) -> Observation[]
        v
observations            <-- APPEND-ONLY SYSTEM OF RECORD (event-grain)
  id, user_id, source, kind, occurred_at,
  subject_identity {kind,value}, object_identity?,
  participants jsonb (full To/Cc/attendees/reviewers + recipientCount + list_id),
  payload jsonb,
  family_key, evidence_hash (unique = (user_id, family_key, evidence_hash))
        |
        |  PROJECTION (deterministic fold; replayable; never the source of truth itself)
        v
  +-- resolve identity ({kind,value} -> stable entity_id; deterministic + reversible merges)
  +-- entity_nodes        (STABLE node id + supersession/forwarding; no projection_version)
  +-- entity_identities   (STABLE typed identity keys, active-unique (user_id,kind,value) WHERE valid_until IS NULL, per-id provenance)
  +-- entity_profiles     (VERSIONED display/kind/significance components + read-time recency)
  +-- entity_edges        (VERSIONED typed: works_at, member_of, reports_to, frequent_collaborator, ...)
  +-- entity_co_occurrence (weighted pair projection; fan-out cutoff; promotion -> typed edge)
  +-- user_facts          (P4-deferred; subject = user | entity_id; key in FACT_ONTOLOGY)
```

Everything below `observations` is derived, but with two different contracts: stable `entity_nodes`/`entity_identities` preserve foreign-key identity and merge forwarding, while versioned read models are **recomputable**. Change a weight, a cutoff, a classifier, or the ontology, and replay the fold into a new projection version without replacing stored entity IDs. Any projection row that becomes Replicache-visible has a third contract: a stable sync key plus a content version that changes only when the serialized value changes, so an active-version flip does not look like delete-all + re-add-all to clients.

## Decisions (resolved in the 2026-06-23 grill)

- **D1 — System of record = event-sourced observation log.** Append-only per-source observations; entities/identities/relations/significance/facts are replayable projections. Observations are retained indefinitely in v1: storage is the price of replay, audit, and future reducer fixes. Superseded observations stop being the active family member, but they are not pruned; only projections are compacted/rebuilt. Rejected: aggregate counters (lossy, unrecomputable), hybrid (two drifting write paths).
- **D2 — Identity = stable entity nodes + a dedicated `entity_identities` table.** Entity IDs are the stable foreign-key surface for the rest of Alfred, so they are **not projection-versioned**. The id is **content-addressed from a single normalized hard identity** via a keyed hash — `entity_id = ent_${base32(hmacSha256(secret, {v:1, userId, identityKind, normalizedValue}))}`. **HMAC, not raw SHA**: emails/logins are guessable and these ids surface in client sync + logs, so the digest is keyed by a server secret (`computeStableEntityId` in `@alfred/db`, secret from `serverEnv`). Never seed from display name, kind, significance, canonical name, a random id, or a projection version — only stable identity material. ("First normalized hard identity" was the earlier wording; rejected as too arrival-order-sensitive for a P0 foreign-key contract.) **Merge survivor = the node seeded by the highest-ranked anchor identity in the merged hard-bridge component** (`identityAnchorRank`, lower wins): (1) user-pinned merge / explicit correction, (2) verified first-party directory identity (Google Workspace), (3) email (the canonical cross-source hard bridge — anchors at this tier whether or not separately "verified"), (4) provider immutable account ids (`github_user_id`, `slack_id`, `notion_user_id`) + org `domain`, (5) provider mutable handle (`github_login`), (6) provisional/source-local. Tie-break after rank: earliest **`entity_nodes.first_seen_at`** (the node's earliest OBSERVATION timestamp, supplied by the writer at mint time via `makeEntityNodeInsert`, never a wall clock — so it is replay-deterministic) → normalized value lexicographic → entity id lexicographic. **Not** newest-row, oldest-row, or whichever source triggered the merge. (The identity rows themselves carry no `first_seen_at` — only the temporal `valid_from`/`valid_until` window; the tie-break is a node property, so it reads `entity_nodes.first_seen_at`.) On merge: move all `entity_identities` in the component to the survivor, stamp `supersedes_entity_id = survivor` on losers, **do not rewrite external FKs synchronously** — reads resolve through `resolveActiveEntityId`; loser ids stay valid forever as forwarding aliases; active projection rows use the resolved survivor id. So `github_login:sanyam-oliv` (seeded id) converges onto `email:sanyam.p@oliv.ai` when the commit-email bridge arrives, with the github-seeded id permanently forwarding. `entity_identities` stores `(user_id, entity_id, kind, value, confidence, source, verified, user_pinned, valid_from/until, supersedes_id, provenance)`, **active-row unique on `(user_id, kind, value) WHERE valid_until IS NULL`** (partial, not global). The partial scope is load-bearing for the table's own temporal columns: mutable+reusable handles (a freed `github_login` reclaimable by another account, a `github_repository_full_name` redirect overridden by a new repo) close the old row (`valid_until` stamped) and open a NEW row for the SAME `(kind, value)` on a DIFFERENT entity — a global unique would force a false cross-entity bridge or block the legitimate re-anchor. So ≤1 LIVE identity holds a `(kind, value)`; closed history may repeat it. Replaces the overloaded `aliases` jsonb; the LIVE set is the dedup index _and_ the join target observations resolve through. Kinds: `email`, `github_login`, `github_user_id`, `slack_id`, `notion_user_id`, `google_directory_id`, `domain` (orgs), `phone`. **Non-person nodes need an anchor too** — `canonical_identity` is NOT NULL on every node and the kind taxonomy (D7) admits `repository`/`project`, so those carry a hard identity that is NOT a person/org key: `github_repository_id` (immutable numeric repo id → provider-account tier), `github_repository_full_name` (`owner/repo`, renamable → provider-handle tier), and `integration_object_key` (generic provider object key for `project` nodes from other sources, the ADR-0062 object-key shape → provider-account tier). Without these, P2 would have to abuse `github_login`/`domain` to mint a repo id and collide semantically with people.
- **D3 — Cross-source merge = deterministic-first + reversible proposals.** Auto-merge only on a hard bridge (shared email; GitHub **commit-author email** from `push` payloads; a confirmed `github_username` fact; verified Google Workspace Directory profile/email when `directory.readonly` + domain sharing are available). Name/fuzzy matches are _proposed_, gated, reversible (propose/dispose, ADR-0062 invariant) — never auto-asserted. Unmapped identities live as provisional entities, merged later via supersession when a bridge arrives. **Coverage is partial by design** (masked `noreply` commit emails don't bridge to work email; fall back to the `github_username` fact).
- **D4 — Observation grain = event-grain, with evidence hashing.** One row per relationship-bearing event version (one email / meeting occurrence / PR-event evidence), full participant set in `participants`. Pairwise edges + significance derived in the fold. A 50-recipient blast is **one** observation carrying `recipientCount: 50` — the reducer down-weights it instead of minting 50 `coRecipient` touches. To keep append-only + supersession coherent, every reducer emits `family_key` (stable event identity, e.g. `gmail:<message_id>`, `calendar:<calendar_id>:<event_id>:<original_start>`, `github:pr:<repo>:<number>`) and `evidence_hash` (only relationship-significant fields: participants/attendees/reviewers/assignees + start where relevant). Idempotency is the unique `(user_id, family_key, evidence_hash)` — no separate `dedup_key` concat is stored (a denormalized `family_key:evidence_hash` would be a second source of truth a reducer could miscompute). Same evidence hash → duplicate/no-op; changed evidence → append a new observation with `supersedes_observation_id` pointing at the prior active family member. Rejected: participant-grain (re-creates today's inflation), bare occurrence-key uniqueness (blocks material changes), etag-only dedupe (trivial edits inflate history).
- **D5 — Graph shape = full social graph, made viable.** Stored as a **weighted pair projection** (`entity_co_occurrence`: one row per distinct co-occurring pair, `weight/count/family_count/last_seen`). **Fan-out cutoff** (`FAN_OUT_CUTOFF = 12`): events above this participant count contribute **zero** pairwise co-occurrence (a blast is not social evidence); below, `pairWeight += SOURCE_WEIGHTS[key] / participantCount` where `key` is the fold-derived interaction class (participantCount = all resolved human-ish participants). **Promotion threshold** (`PROMOTION_THRESHOLD = 2.0`): a pair becomes a traversable typed `frequent_collaborator` edge only past this weight bar; below that it's queryable data, never traversed. **Promotion guardrails** (`PROMOTION_MIN_OBSERVATIONS = 3`, `PROMOTION_MIN_FAMILIES = 2`): a pair must also be backed by ≥3 distinct observations across ≥2 distinct event families — `MIN_FAMILIES` is load-bearing, else one long email thread or one chatty PR accidentally promotes. So traversal only ever walks typed + promoted edges (small, indexed). Full graph, bounded size, near-linear writes, recomputable. All four constants are locked v1 values in `@alfred/contracts` (text/registry knobs, no migration to tune).
- **D6 — Significance = deterministic source-weighted fold + read-time recency.** Forced deterministic by the event-sourced choice (can't replay an LLM over all history per weight change). The projection materializes only **time-invariant components** (`volume`, `reciprocity`, `sameOrg`, source-weighted interaction weights, raw co-occurrence weight, top contributing observations); `lastSeenAt` is a **first-class sibling column** on `entity_profiles`, deliberately NOT inside the significance components — it is the wall-clock input to the read-time recency multiplier, and folding it into the checksummed time-invariant set would break D13 replay determinism (it was moved out of `significanceComponentsSchema` for exactly this reason). The locked v1 `SOURCE_WEIGHTS` (`@alfred/contracts`): `github_review 1.0`, `calendar_meeting 0.9`, `gmail_reply 0.8`, `gmail_direct 0.65`, `gmail_cc 0.25`, `github_push 0.25` (strong object/repo signal, weak person↔person), `gmail_blast 0.0` (the fan-out cutoff already zeroes its co-occurrence). **P1 can only calibrate the Gmail weights against prod — `github_*` and `calendar_meeting` stay provisional until P2/P3 shadow validation.** The final score is computed at read time as `baseScore(components) * recencyMultiplier(asOf, lastSeenAt)` (e.g. exp decay), so deterministic projection checksums do not change as wall-clock time advances and no daily full re-projection is needed just for decay. One score per user (persona is already separated — work vs personal are distinct `user` rows).
- **D7 — Kind taxonomy, classified into the versioned profile.** `person | organization | group | service | repository | project`. Detection: [`List-Id`](https://datatracker.ietf.org/doc/rfc2919/)/`List-Unsubscribe` headers (authoritative) + local-part heuristics (`team@`, `all@`, `*-eng@`) + fan-out for `group`; `noreply@`/`notifications@`/`*-mirror@` for `service`; GitHub repo addresses → `repository` (linked `in_org`, not `works_at` as a person). Non-humans are **retained as typed nodes** (queryable, recomputable, no signal lost) but **never person-significance-scored**. Because better classifiers can change `kind`, it lives in versioned `entity_profiles`, not on the stable entity node. Rejected: suppress-at-ingest (loses signal), is_automated flag (conflates repo/list/bot).
- **D8 — `user_facts` = typed ontology + subject binding.** Every fact binds to a **subject** (`{kind:'user'}` or `{kind:'entity', id}`) and a `key` from a **registered durable fact-type** (`FACT_ONTOLOGY` in `@alfred/contracts`): `employer, job_title, manager, reports_to, timezone, github_username, …`. Facts about people hang off their entity node (the "personalized relevance" destination). Transient document content (passcodes, alarm names, incident times) **is not a fact-type** → stays in the document / a `memory_chunk` (RAG), never auto-confirmed as a user fact. Rejected: TTL/decay-only (leaves the junk drawer, no subject).
- **D9 — `integration_object_relations` will be wired through the resolver, storing stable entity IDs.** ADR-0062's object/entity separation stays (work-objects out of the canonical people table). In P2, each GitHub webhook emits **two** projections: observations into the social/significance graph (a review _is_ a work-relationship signal) **and** the `authored_by`/`reviewed`/`assigned`/`closes` object↔entity edges via the identity resolver. `integration_object_relations.entity_id` still references the legacy `entities` table in P0; P2 is the cutover to stable entity IDs + supersession resolution so D13 projection flips cannot dangle object links.
- **D10 — One-time substrate migration = shadow-build + validated cutover.** New tables coexist with the legacy aggregate graph. Replay all sources into the observation log; project the new graph; **validate coverage against the old graph** (row counts, expected identities present, no source family dropped) while treating legacy significance ordering as a polluted baseline that is expected to diverge. Correctness comes from deterministic replay checks and spot-checks: `engineering@`→group, repos→repository, Sanyam GitHub↔email merged where a hard bridge exists, flat-0.368 standup list differentiated. Flip consumers behind a flag, watch, then drop legacy columns/tables. Idempotent by the unique `(user_id, family_key, evidence_hash)`; dry-run locally; prod replay via `railway ssh`. Rejected: in-place rebuild (empty-graph window on the account you depend on), forward-only (wastes history in `documents`/`webhook_events`).
- **D11 — Sequencing = ADR + plan first, then incremental Gmail-first.** Each phase shadow-builds and validates before the next. Gmail-first because it's the source with the most data/intuition — cleanest validation of the new projection before adding sources.
- **D12 — Governance = this epic is the foundation; ADR-0066 consumes it.** No two systems on the same tables. The triage rewrite reads the new graph.
- **D13 — Steady-state projection re-versioning is checkpointed and read through active views.** Event-sourcing only pays off if replay is deterministic. Each projection records `(projection_name, projection_version, source_high_watermark, checksum, completed_at)`. Only recomputed outputs are projection-versioned: `entity_profiles` (display/kind/significance components), `entity_edges`, `entity_co_occurrence`, and P4 `user_facts`. Stable `entity_nodes` and `entity_identities` are not version-partitioned because external tables store `entity_id`. A weight/ontology/reducer change bumps a version, shadow-replays, compares checksums/counts/top-N at a fixed validation `asOf` when final scores are involved, then atomically flips the active projection pointer. Consumers do **not** hand-write `WHERE projection_version = active`; they read `active_entity_profiles`, `active_entity_edges`, `active_entity_co_occurrence`, `active_user_facts`, etc. views that pin the current version and join to stable entities. Checksums are over deterministic row order (`ORDER BY id` / stable key), rounded floats, and **time-invariant components only**; final recency-decayed scores are excluded because they are read-time values. Rejected: "just rerun the script" with no replay identity; projection-versioning stable IDs (D13 flip would dangle `integration_object_relations`/facts/todos/briefings); shared versioned rows without active views (one forgotten filter gives mixed-version reads).
- **D14 — User actions are source events too, with rank-then-recency precedence.** Chat-captured standing instructions, user confirmations/rejections, manual profile edits, and correction feedback enter the same log as high-precedence `source='user'|'alfred_chat'` observations. The fold resolves conflicts by source rank first (`user` > `alfred_chat` > first-party integrations > enrichment/RAG), then recency within the same rank. Projections may propose facts from integrations, but they must never overwrite user-authoritative corrections; they add superseding observations or proposed conflicts. Existing non-proposed `user_facts` rows (`confirmed`, user-edited, rejected/dismissed, and `standing_instruction`) migrate into `source='user'` observations, then project back into `user_facts`. Status/confidence must be derivable from the fold: a confirmation/edit/rejection is itself a user observation, so replay cannot downgrade confirmed facts back to proposed.
- **D15 — Reducers distinguish raw events from relationship evidence.** A provider event can produce several observations, but only relationship-bearing occurrences affect significance/co-occurrence. Example: a Calendar event update or reminder change should not count as another meeting with the attendees; a meeting occurrence does. GitHub push/PR/review events similarly need reducer-specific evidence kinds so significance is not inflated by retries, edits, or noisy automation.
- **D16 — Stable IDs are the foreign-key contract; versions are read-model contracts.** Anything outside the projection engine (`integration_object_relations`, `user_facts.subject`, future todos/briefings/meeting-preps) stores only stable `entity_id`s. Re-versions may change kind, significance, relations, and co-occurrence, but they must not mint replacement entity IDs. Merges choose a deterministic winner, write forwarding on losers, and every read path resolves through the forwarding chain before joining active projections.
- **D17 — Replicache-visible projections need stable sync keys + content versions.** Today's sync diff is CVR over per-row `rowVersion`; a projection-version flip that swaps physical rows without stable keys would look like delete-all + re-add-all and can wipe optimistic local state. Every active view that feeds Replicache (`active_user_facts`, and any future `active_entity_profiles` surface) must expose a stable logical key and a synthetic `rowVersion`/content version derived from serialized-content hashes. The projection cutover computes per-key added/changed/removed deltas: unchanged keys keep their prior sync version, changed keys bump once, removed keys delete, new keys insert. This belongs in P0 schema/read-model design and P5 cutover validation, not a later UX cleanup.

## New schema (additive; `packages/db/src/schema/`)

> All tables carry `user_id` (CASCADE per single-user FK convention), `lifecycle_dates`, and `createId` prefixes. Names indicative; finalize in P0.

- **`observations`** — `id (obs)`, `user_id`, `source`, `kind`, `occurred_at`, `family_key`, `evidence_hash`, `subject_identity jsonb {kind,value}`, `object_identity jsonb?`, `participants jsonb`, `payload jsonb`, `schema_version`, `reducer_version`, `supersedes_observation_id?`, `lifecycle_dates`. Indexes: `(user_id, source, occurred_at)`, `(user_id, family_key)`, unique `(user_id, family_key, evidence_hash)`. (No stored `dedup_key` concat — a denormalized `family_key:evidence_hash` would be a second source of truth a reducer could miscompute.)
- **`entity_nodes` (stable)** — `id (ent)` content-addressed from `(user_id, normalized hard identity)`, `user_id`, `canonical_identity jsonb`, `supersedes_entity_id?`, `first_seen_at`, `lifecycle_dates`. No `projection_version`. External references store this ID after the P2/P5 cutovers; read helpers resolve forwarding chains.
- **`entity_identities` (stable)** — `id (eid)`, `user_id`, `entity_id` → stable `entity_nodes`, `kind`, `value`, `confidence`, `source`, `verified bool`, `valid_from/valid_until`, `supersedes_id`, source-observation provenance, `lifecycle_dates`. Active-row unique `(user_id, kind, value) WHERE valid_until IS NULL` (partial — closed history may repeat a mutable handle that was freed/reused; ≤1 LIVE row holds a `(kind, value)`); index `(user_id, entity_id)`. `aliases` jsonb is retired in favor of this table (kept during shadow phase, dropped at cutover).
- **`projection_runs` / `projection_cursors` / `active_projection_versions` / `projection_sync_state`** — tracks active projection versions, source high-watermarks, row-count/checksum summaries, completion status, cutover pointer, and Replicache-visible sync state `(sync_slug, stable_key, content_hash, row_version)`. This is the replay safety rail; consumers read active-version views, never unfiltered projection tables.
- **`entity_profiles` (versioned)** — `id (eprof)`, `user_id`, `projection_name`, `projection_version`, `projection_run_id` → `projection_runs`, `entity_id` → stable `entity_nodes`, `display_name`, `kind`, `significance_components jsonb`, `last_seen_at`, `provenance jsonb`, `computed_at`, `lifecycle_dates`. Unique `(user_id, projection_name, projection_version, entity_id)`. Composite run FK `(user_id, projection_name, projection_version, projection_run_id) → projection_runs(user_id, projection_name, projection_version, id)` so the row's `projection_name` + `projection_version` are bound to its run's — `projection_runs` is generic (P4's `user_facts` projection reuses it), so without the name a foreign-projection run could occupy the unique slot or be named by the FK. Provenance + delete-by-run cleanup, not a free-floating tag. Read helpers expose final recency-adjusted score from components + caller `asOf`.
