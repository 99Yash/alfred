# Alfred — Shared Vocabulary

Living glossary of the load-bearing terms. One to three lines per term. Refine in place when usage drifts; don't re-litigate in chat.

Cross-references: [`decisions.md`](./decisions.md) (the ADRs, snapshot table at the top), [`CLAUDE.md`](./CLAUDE.md) (operational guide).

---

## Core domain

**Skill.** A markdown body + frontmatter the agent mounts into its system prompt. Activated by `@skill:slug` in a brief or chat. Authoritative storage: `skills` + `skill_revisions`. ADR-0017.

**Workflow (row).** A row in the `workflows` table. Holds trigger spec, brief, optional steps DAG, status, allowed-integrations, HIL gates. Source of truth for the _settings UI_ and the _trigger dispatcher_ — not for execution shape.

**Workflow (code).** A `Workflow<S>` object passed to `registerWorkflow()` at server boot. Source of truth for _execution_ (steps, initialState, dedupKey). Built-ins live as both a row AND a code object; user-authored live as a row only — registry misses route through a single shared `userAuthoredBriefWorkflow` sentinel only after the checked run resolver verifies `workflows (userId, slug)` exists and `is_builtin=false`. The requested slug, not the sentinel slug, stays the join key on `agent_runs.workflow_slug`; execution shape is shared.

**Run.** One execution of a workflow. Stored as one row in `agent_runs`. Joins back to the workflow via `agent_runs.workflow_slug`. There is no separate `workflow_runs` table — `agent_runs` covers status, timing, cost attribution.

**Brief.** Two scopes, both legitimate:

1. _Workflow brief_ (ADR-0017): the user's natural-language description of what the workflow should do. Stored on `workflows.brief`.
2. _Sub-agent brief_ (ADR-0026): the initial transcript handed to a spawned sub-agent. Stored as the first message of a child `agent_runs.transcript`.
   When ambiguous, qualify: "workflow brief" or "sub-agent brief."

**Trigger.** Discriminated union on `workflows.trigger.kind`: `cron` | `event` | `manual` | `on_signal`. The dispatcher consults `status='active'` before enqueuing a run.

**HIL gate.** A step id listed in `workflows.hil_gates`. The runtime parks the run on `wakeCondition.kind='hil'` when entering the step; user approval flips it back to `runnable`. Only meaningful with explicit `steps`. For brief-only workflows (m13+), HIL is driven by the **user action policy** instead — see below.

**User action policy.** Per-user row in `user_action_policies` storing `default_mode` (`autonomy` | `gated`), `integration_rules` jsonb keyed by integration slug, and `approval_notify_delay_ms`. The tool dispatcher consults it on every tool call; `gated` results land as staged actions awaiting human decision. Default at signup: `gated`, with `system.*` tools seeded to autonomy. ADR-0034. **Editor (m13 Phase 8c):** a per-integration **autonomy / gated** radio on the `/integrations/$provider` detail page (per-tool overrides + `default_mode` editing deferred from the v1 UI). The row is **Replicache-synced** (single entity keyed by `userId`; the web derives each integration's mode from `integration_rules[slug].mode ?? default_mode`), so it gains a `row_version` column + bump. The policy mutation does **two** invalidations: bump `row_version` (web pull) **and** `publishPolicyBust(userId)` (in-process dispatcher cache across instances). ADR-0034 amendment 2026-05-29.

**Policy mode.** `autonomy` (execute immediately) or `gated` (stage for HIL approval). Resolution at dispatch: **run-scoped auto-mode override → per-tool override → per-integration mode → user default**. Both modes are const-narrowed string unions from `@alfred/contracts`, never open strings. **`gated` gates reads too** — there is no read/write split and no carve-out for the trigger's own document: under a gated integration the boss stages even `gmail.read_message`, including reading past the inline 4k `<trigger_event>` excerpt of the email that fired the run (`riskTier` is a UX hint, never a gating shortcut). The 4k excerpt is the ungated baseline; per-tool/per-action overrides (deferred, ADR-0034) are the future relief valve. This behavior must be surfaced in UX copy at credential consent / on the integration detail page so a user choosing `gated` knows reads will park — see m13 plan deferred follow-up.

**Auto mode (chat).** The composer's `autoMode` toggle ("auto mode" / "manual review", `dimension-chat-thread.tsx`). "Auto mode" = a per-thread blanket `autonomy` override captured onto each run and applied at the top of policy resolution; "manual review" = honor the durable `user action policy`. **Server-authoritative** once a run exists (the gate runs server-side and background runs have no browser). The eventual thread/conversation row stores the per-thread default; `localStorage`/global store holds only the new-chat default toggle position (default **manual**) until first send persists it. No riskTier carve-outs — auto mode is full autonomy for the thread (ADR-0034 alt-(f)). Wiring rides the m13 chat→runtime bridge. ADR-0034 amendment 2026-05-27.

**`@alfred/contracts`.** New tiny package (sibling to `@alfred/sync`, `@alfred/env`) holding cross-boundary types + const tables: `INTEGRATION_SLUGS`, `POLICY_MODES`, per-integration `*_ACTIONS` lists, derived `ToolName`, future attribution/signal kinds. Zero Node deps; importable from `packages/db`, `packages/api`, `apps/web` without runtime-bundle leaks. Pure named exports + `as const`; no side effects at import time.

**Tool name.** Canonical form `${IntegrationSlug}.${ActionSlug}` — both halves are const-narrowed unions from `@alfred/contracts`. Per-integration action lists compose via `INTEGRATION_ACTIONS` into a single `ToolName` template-literal type. `system.*` is for internal tools such as exact discovery/loading, current time, and scratchpad operations. Schema columns and dispatcher signatures use `ToolName` exclusively; no open strings, no `string & { __brand }` shortcuts.

**Action staging.** Row in `action_stagings` — one per proposed tool call, gated or autonomy. Gated rows park the run on `wakeCondition.kind='hil'` with `approvalId=stagingId` until user decides (approve / edit / reject); autonomy rows transit `pending → executed` in ms. Idempotent on `(run_id, tool_call_id)` for crash-resume. Carries `row_version` for Replicache, `risk_tier` for UI/email snapshots, and `proposed_input_hash` for retry suppression. Canonical audit record for "what did Alfred try and what happened?" ADR-0034. **Card provenance projection (m13 Phase 5 UI):** the `ACTION_STAGING` Replicache fetcher enriches each synced row with read-only, derived provenance — `workflowName` (joined from `workflows.name`, falls back to slug), a **narrowed display-only `trigger`** projection (`{ kind, source?, type? }`, never the raw payload/document ids), and a server-truncated `brief` (~280 chars) — so the approval card can render "what is this / where did it come from" without a separate run-detail fetch.

**Approval debounce.** When a gated staging row lands, SSE pokes the web UI immediately; a BullMQ delayed job is also scheduled using `user_action_policies.approval_notify_delay_ms` (default 5min). If the user decides in-app before the delay fires, the job is removed and no email goes out. If the job fires with the row still `pending`, it calls `notify({ userId, kind: 'approval', idempotencyKey: 'approval:' + stagingId, ... })`, sending one email per staging row. Tracked by `action_stagings.notify_after_at` (scheduled fire time) and `notified_at` (actual fire time). ADR-0034.

**Approvals surfaces.** Two distinct read models. **(1) Live pending queue** (`/approvals`): Replicache-synced (`status='pending' AND requires_approval`), bounded (rows auto-expire at 24h), so pagination + filtering (by integration + risk) happen **client-side** over the synced collection — filter state lives in URL search params; "pagination" is windowing, not server paging. No server query endpoint for this surface. **(2) History** (resolved actions — approved/rejected/expired): _not_ synced, grows unbounded, so it's a separate **server-paginated + filterable** read model (`GET /approvals?status=&integration=&risk=&page=`). **Deferred** (grilling 2026-05-31) — built when the History tab lands. Decided during the Phase 5 approvals-UI polish.

**Rejection contract.** When the user rejects a staged action, the boss's resumed turn receives a structured tool-result: `{ status: 'rejected_by_user', toolName, proposedInput, reason, retryPolicy: 'do_not_retry_identical' }`. The dispatcher _enforces_ the retry policy: a second stage attempt with the same `(run_id, tool_name, hash(proposed_input))` synthesizes another rejection without re-staging or re-emailing. Reject UI exposes two affordances: "Reject and continue" (default) and "Reject and end run" (triggers `cancelRun(runId)` with `reason='cancelled_by_user'`). Reason text is required. Staging cards render a banner with the most recent rejection for the same `(user_id, tool_name)` if one exists in the last N days — so the user (and the boss, on resume) sees why a similar action was rejected before. ADR-0034.

**Tool risk tier.** Const-narrowed enum at tool registration time, declared in `@alfred/contracts`: `no_risk | low | medium | high`. Pure UX hint at v1 — the dispatcher reads only `user_action_policies`, never `riskTier`. Drives integration-card summaries ("Gmail — 12 tools (4 high, 5 medium, 3 low)"), staging-card badges, and email subject prefixes. Staged rows snapshot the risk tier so old approvals don't change copy after a registry edit. Authors classify on registration; trust-the-author at v1, custom lint rule when the registry grows past ~30 tools. ADR-0034.

**User context.** Compact, queryable runtime profile about the user: bio, preferences, current company/projects, important people, known contacts, relationship facts, and eventually saved third-party dossiers. Source of truth stays in Postgres (`user_facts`, `user_preferences`, `entities`, `entity_relations`, `memory_chunks`, later `person_profiles`); Redis may cache a derived slice such as `alfred:user-context:{userId}:v1` for fast reads, but cache loss must only cause a rebuild. Boss/sub-agents access it through `system.read_user_context`, an always-available system tool rather than a loadable `memory` integration. ADR-0042 amendment.

**Write surface.** The set of write/mutating tools (`gmail.send_draft`, `calendar.create_event`, `docs.create`, `drive.share`, …) an integration exposes. First-class as of ADR-0043 — _not_ walled off architecturally. Authorization is a three-layer composition: tool registry (does the tool exist?) → declared tool surface bounded by `workflows.allowed_integrations` and connected credentials → `user action policy` (`autonomy | gated`, default `gated`). As of ADR-0053 the middle layer is **dispatch-enforced**, not exposure-only: `dispatchToolCall` hard-checks the `allowed_integrations` cap + scope-aware connection health before executing any registered tool (previously a qualified call could bypass the cap). Supersedes ADR-0033's "never expand the write surface" rule, which now scopes only to the tool-free briefing compose path. Safety moved from architecture to the policy default. ADR-0043 + ADR-0053.

**Publishing posture.** Google OAuth consent screen runs in **Production, deliberately unverified** (not Testing). Avoids Testing's 7-day refresh-token expiry for non-profile scopes; refresh tokens are still revocable and subject to Google's normal token limits. The trade is the unverified-app warning + 100-new-user cap, both fine at single-tenant scale. We never submit for verification, so "Google won't approve it" doesn't apply on this path. ADR-0044.

**Scope tier.** Google's three-way classification that sets the verification bar: **non-sensitive** (`drive.file`, `openid`, `email` — no sensitive/restricted scope review), **sensitive** (`gmail.send`, `calendar.events`, Workspace edit scopes — app verification, no security assessment), **restricted** (`gmail.modify`, full `drive` — restricted verification **+ annual security assessment if restricted data is stored/transmitted through servers**, the real public wall). Alfred requests least-privilege, extends freely into sensitive, and takes exactly one restricted scope (`gmail.modify`) because reading mail is the product. Granted set tracks the registered tool set. ADR-0044.

**Multi-tenant-capable / single-tenant operation.** Alfred is _architected_ multi-tenant (per-user `integration_credentials`, `user_action_policies`, `user_id` partitioning) but _operated_ as one tenant. "Single-user" (ADR-0001) is the current operating mode, not an architectural ceiling — going public is a verification submission + ADR-0009 allowlist removal, not a rewrite. A self-host-per-user model (each user = their own GCP project) is the open third path that avoids a centralized public OAuth client and its restricted-scope security assessment. ADR-0044.

## Compaction

**Transcript compaction.** Distinct executor step inserted between `dispatch-tools` and the next `boss-turn` when token-count crosses `compactionThresholdTokens(Math.min(bossWindow, compactorWindow))` (60% of the _smaller_ of the boss and compactor context windows, both from `model_prices.context_window`). Compactor is **Claude Sonnet 4.6 with thinking disabled** (a `COMPACTOR_MODEL` constant — not `getCheapModel()`, not a new tier dispatcher), fallback **Gemini 2.5 Flash** (1M window); output capped at 2000 tokens. A pre-call guard fails over to the fallback (or `compactor_input_too_large`) if the prior slice exceeds the compactor's window. ADR-0035 amendment (2026-06-01) — see that ADR for the rare-but-quality-critical rationale and the `min()` requirement under asymmetric boss/compactor windows. The stable boss system prompt and tool definitions stay outside `agent_runs.transcript` as `AlfredAgent.turn()` inputs; the compactor preserves the in-flight tail (last assistant message + its tool calls + results) verbatim and compresses everything older into the run handoff. Sub-agents do not compact (fail back to the boss for re-decomposition per ADR-0026). The compactor itself is exposed as a reusable primitive (`compactTranscript({ prior, inFlightTail, attribution })` in `packages/api/src/modules/agent/compaction/`) so a future long-running chat surface can call it without copying the boss workflow's policy. Background-agent workflows (triage, briefing, cold-start) don't loop and don't compact — the primitive only fires from looping agent drivers that cross the threshold. Compactor failure triggers a **bounded in-step retry** (3 attempts, 100ms then 200ms backoff) before the run fails with `compactor_failed: <reason>`. ADR-0035.

**Run handoff.** Structured XML `<run_summary>` emitted by the compactor, replacing older transcript content. Sections: `goal`, `user_directives`, `decisions`, `actions_completed`, `actions_rejected`, `actions_failed`, `sub_agent_findings`, `pending_followups`, `key_entities`. A third ephemeral `cacheControl` breakpoint goes after the `<run_summary>` so subsequent turns hit a stable cached prefix until the next compaction. ADR-0035.

**User directives vs decisions.** Two distinct handoff sections by design. `<user_directives>` = pragmatic, mid-run intent statements that bound the agent's future behavior ("trust gmail for the rest of this conversation"), preserved verbatim. `<decisions>` = epistemic, facts/preferences/constraints learned during the run ("Alice is the manager"). The compactor's system prompt enforces "do not paraphrase under `<user_directives>`" — paraphrasing introduces drift on intent grants. **Supersession:** when a later directive revokes/overrides an earlier one, both are kept verbatim in chronological order and the earlier is tagged `<directive superseded="true">` — the marker is metadata (not a paraphrase, so "verbatim" holds), and the boss acts only on non-superseded directives. ADR-0035 (+ amendment 2026-06-01).

## Scratchpad

**Run scratchpad.** Per-run K/V store for sub-agent findings + boss-promoted shared state. **Redis** is the live store during the run (keys `alfred:scratch:{runId}:{zone}.{path}`, 30-day TTL). **Postgres** receives a per-key snapshot via `agent_run_context` at the executor's terminal step (success / failure / cancel), idempotent via `ON CONFLICT (run_id, key) DO UPDATE`. Crash-resume of a lost scratch entry mid-run = re-execution of the producing step (ADR-0014's idempotency). Per-zone single-writer enforced at the dispatcher: only `sub_a` writes `scratch.sub_a.*`; only the boss writes `shared.*`. ADR-0036 (amends ADR-0016).

**`shared.*` vs `scratch.{subId}.*`.** Two namespaces inside the run scratchpad. `scratch.{subId}.*` is sub-agent advisory state (unvalidated; the sub-agent owns it). `shared.*` is boss-promoted canonical state (validated; only the boss writes). Sub-agents read both but write only their own zone. Cross-pollination across sub-agents flows through `shared.*` — the boss is the gate. ADR-0016 + ADR-0036.

**Active tool surface.** `agent_runs.state.activeTools: ToolName[]` is the exact run-local model and execution surface. Global registry membership is server capability only. Dispatch rejects disallowed tools before activation; an allowed inactive call activates its exact schema and bounces the model to issue a fresh call rather than validating schema-blind arguments. Legacy `activeIntegrations` checkpoints expand once into exact integration tools plus the system kernel. The always-visible kernel (`availability.surface: "kernel"`) is the discovery pair (`system.search_tools`, `system.load_tool`), `system.current_time`, and the four tools the chat/boss prompt names by name — `system.read_user_context`, `system.web_search`, `system.read_chat_history`, `system.spawn_sub_agent` (7 tools; a tool the prompt advertises must be eager or its first use pays a search/load dance plus a mid-run prompt-cache bust). The kernel is returned unfiltered, so the `activeTools → ToolSet` projection (`buildSdkToolSet`) drops tools the caller can't invoke — `requiresThread` (chat-only) and `callers` (boss-only) — leaving 6 in thread-less briefs and 5 in sub-agent runs. Every other system capability is discovered/preloaded/loaded exactly. ADR-0053 amendments / #407 / #411 / #412 (PR #519).

**Connected summary.** A run-start snapshot of connected ∩ allowed integrations, rendered into the boss/chat preamble as one line per integration (`slug — actions — short desc`, with `(needs reauth)` for unhealthy credentials). In v1 it is **grounding copy**, not the only visibility channel: full schemas for usable connected ∩ allowed tools are declared eagerly. Skip empty-action stub integrations. Source: `integrationCredentials` + slug-specific scope checks. Snapshot into run state; the system resolver only concatenates the frozen string. ADR-0053.

**Connection health.** Per-integration usability, from `integrationCredentials.status` (`active` vs `needs_reauth`/absent) **plus slug-specific OAuth scopes**. One shared helper answers "is this integration usable right now?", consumed by the connected summary and dispatcher gate, returning structured `needs_connect` / `needs_reauth` / `not_allowed` instead of cryptic provider failures. Google fan-out is scope-aware: a Calendar-only Google credential does not make Sheets/Slides/Drive usable. Multiple Google rows reduce by "any active row with the required slug scopes wins; otherwise needs_reauth if a relevant row exists but is unhealthy/missing scopes; otherwise needs_connect." ADR-0053.

**Dispatch floor.** Plain code in `dispatchToolCall` that, before execute, (a) resolves bare→qualified tool names and returns `ambiguous_tool` for collisions like `batch_update`, and (b) **hard-enforces** `allowed_integrations` + scope-aware connection health (closing ADR-0043's exposure-only cap hole). The result returned to the boss is always an actionable `{status, message, ...}` envelope. No v1 auto-activation path is needed because connected ∩ allowed tools are already declared eagerly. ADR-0053.

**Run grounding.** The run-start snapshot string concatenated into the boss/chat system prompt by the `AlfredAgent` system resolver — cache-stable by construction (no live DB/health reads mid-turn; ADR-0053 + ADR-0026). **Members:** _date grounding_ (today's date + `IanaTimezone`, via `formatDateGrounding`), the _connected summary_ (ADR-0053), and _standing instructions_ (the bounded set of durable behavior rules — ambient because they bias every turn; relevance-filter if the set grows large). **Standing instructions are dual-delivered:** ambient via Run grounding for the boss/chat, **and pulled by the non-boss pipelines** (triage `classify`, briefing `gather`) via a dedicated read path — those pipelines never see Run grounding, so a directive that must bias them (e.g. a sender suppression) is consulted at classify/gather time, not injected into a prompt they don't read (see _Suppression standing instruction_). It is the read-surface of ambient, always-true facts. Facts + people are **not** ambient — they stay pull-on-demand via `read_user_context`/`recallMemory`. **Distinct from** _tool-selection recovery_ (the dispatch-floor `unknown_tool` envelope — a recovery concern, not grounding) and from _user context_ (the durable memory read via `system.read_user_context`). The boss perceives the world through three channels: **run grounding** (ambient prompt facts), **declared tool schemas**, and **`read_user_context`** (durable memory). GROUND-001 in [docs/plans/june-demo-triage.md](./docs/plans/june-demo-triage.md).

**Builtin vs user-authored.** `workflows.is_builtin = true` for alfred-curated workflows seeded from the repo (immutable except `status`); `false` for user-authored (full CRUD). Same table, same toggle UX.

## Runtime primitives

**Tick.** A BullMQ repeatable that fans out per-user. Existing pattern: `briefing.tick` (hourly, `packages/api/src/modules/briefing/repeatable.ts`). m12 adds `workflows.tick` (every minute, generic) per ADR-0027.

**Dispatcher.** The piece that turns a trigger into a `createRun` + `enqueueRun` call. Exists implicitly per-feature today (e.g. the briefing tick handler); m12's `workflows.tick` is the first generic dispatcher. Cron fires via the tick; `event` fires via the `emitEvent` bus (below).

**Event bus (`emitEvent`).** The generic `event`-trigger dispatcher (m13 Phase 8a). `emitEvent({ userId, source, type, eventId, payload })` queries `workflows WHERE status='active' AND trigger.kind='event' AND trigger.source=… AND trigger.type=…` and `createRun`s each match with a uniform `input: { documentId, reason, source, type }` + `trigger: { kind:'event', source, type, eventId, payload: { documentId, reason } }`. **One dispatch path:** the email-triage builtin is matched like any other event workflow — `enqueueTriageRuns`' hardcoding is deleted; the Gmail ingestion worker calls `emitEvent` per freshly-inserted doc instead. Triage's `initialState` still reads `input.documentId/reason` (run behavior byte-identical); only its dispatch trigger changed from imperative to query-matched. ADR-0047 (extends ADR-0027).

**Event source + type.** Closed const-narrowed pair in `@alfred/contracts` (`EVENT_SOURCES`, per-source `*_EVENT_TYPES`), replacing the old informal `source: 'gmail.ingest'` string. Shipped set: `gmail.message_received`, `google.oauth.callback.completed`, `learn-skill.completed` (ADR-0047 amendment 2026-05-31 — gmail + oauth-callback are live producers; learn-skill's bus emit is in flight). An enum value may land one milestone ahead of its producer (a typed const, unlike the `on_signal` _subscriber_ that was deferred), but no source ships without a committed producer. The `webhook|ingest|manual` breadcrumb rides `trigger.payload.reason`, not the type. `trigger.filter` stays on the schema for forward-compat but the v1 dispatcher does **not** evaluate it (API rejects non-empty `filter` writes, like `steps`); the brief-only boss decides relevance itself. ADR-0047.

**`<trigger_event>` message.** For an `event`-triggered brief-only run, the sentinel's `initialTranscript` (async + DB-aware, with `userId` and `trigger`) appends a `user`-role `<trigger_event>` message after the brief carrying a **bounded trigger context** resolved from `documents`: ids, source/type, title/subject, sender metadata, authored time, URL, and a raw content excerpt capped at 4,000 characters for v1. Inbound third-party content is never blindly inlined in full; if the boss needs more, the auto-seeded source tools let it call the registered provider read tool (e.g. `gmail.read_message`). User-authored outbound content may be inlined in full when Alfred itself created it. Raw bodies are **not** persisted in `agent_runs.trigger.payload` — the payload stays a pointer + breadcrumb. Missing/deleted documents degrade to an unavailable trigger message instead of failing run creation. Event runs auto-seed the trigger source's exact registered tools into `activeTools` (∩ `workflows.allowed_integrations`, same cap rule as `@`-mentions) so a Gmail-triggered workflow has Gmail tools without a redundant `@gmail`. ADR-0047 (extends ADR-0040; amended by #407).

**Event idempotency (bounded).** No DB unique index on `eventId`. The Gmail worker fans out only _freshly-inserted_ `documentIds` (insert-once → emit-once) behind the webhook handler's existing 30s TTL dedup, and `emitEvent` does a best-effort non-terminal duplicate check on `(userId, workflowSlug, source, type, eventId, reason)` before creating a run so a retried ingestion job does not normally double-fire. This check has a race and is not a guarantee. A hard `(user_id, workflow_slug, eventId)` constraint is explicitly rejected — triage reuses `eventId=documentId` across `reason`s (e.g. a later reply-retriage), which such an index would wrongly block. ADR-0047.

**`next_run_at`.** Denormalized timestamp on the `workflows` row. Recomputed (via `cron-parser`) at exactly two moments: when `trigger`/`status` mutates, and inside the tick after a successful `createRun`. The tick query is an index lookup, not a scan. ADR-0027.

**`trigger` (on `agent_runs`).** First-class jsonb column on the run, mirroring `workflows.trigger.kind` plus per-kind metadata (`scheduledFor`, `eventId`, `payload`, `signalName`). Source of truth for "why did this run fire?" Replaces ad-hoc `metadata.triggeredBy` stuffing. ADR-0027.

**Scheduled-instant jobId.** BullMQ `jobId = workflow:{workflowId}:scheduled:{nextRunAtIso}`. Idempotency primitive: a retried tick is a no-op via BullMQ's native dedup without consulting Postgres. ADR-0027.

**Per-turn LLM driver.** `AlfredAgent` (ADR-0026, `packages/ai/src/agent.ts`). One `turn()` call = one LLM round-trip = one `api_call_log` row. Composes with the durable runtime; not yet wired into any agent_run.

**Call role.** `CallRole` union (`packages/ai/src/metering/types.ts`): `'compactor' | 'boss' | 'sub_agent' | 'triage' | 'briefing' | 'cold_start' | 'memory_extraction'`. Set on `attribution.role` when calling `meteredGenerateText` / `meteredGenerateObject`; surfaces on `api_call_log.request_meta.role` so a single run's spend can be split between agent surfaces (boss vs sub-agent vs compactor) without adding a column. Phase 7 wires all three boss-workflow roles end-to-end: `'boss'` and `'sub_agent'` on `AlfredAgent.turn()` inside `userAuthoredBriefWorkflow`, `'compactor'` on `compactTranscript`. The remaining roles (`'triage'`, `'briefing'`, `'cold_start'`, `'memory_extraction'`) are typed but not yet wired at their call sites — that's a separate small PR. Query with `request_meta->>'role'` in cost rollups.

**Brief-only execution shape.** Two named executor steps that ping-pong: `boss-turn` runs exactly one `AlfredAgent.turn()` (preserves ADR-0015's one-turn-one-`api_call_log` invariant); `dispatch-tools` calls `dispatchToolCall` for each returned tool call, appends results to the transcript, and either returns `next: 'boss-turn'` or `interrupt` if any call staged for HIL. Compaction (Phase 7) slots in as a third named step. The shared code-side `Workflow<S>` handles every user-authored brief — the slug on `agent_runs.workflow_slug` distinguishes runs, not their execution shape.

**System-tool dispatch contract.** `system.*` tools are registered with the same `liveTool` factory as integration tools, but the dispatcher applies a **structural autonomy override**: `if (integration === 'system') policyMode = 'autonomy'`, ahead of the `user_action_policies` lookup. State-changing system tools such as `spawn_sub_agent` take the full dispatcher path; chatty no-op-side-effect scratchpad tools get a fast-path that skips staging. Tool `execute` remains pure. `system.load_tool` returns an exact successful name, and the `dispatch-tools` step applies that bounded effect to `activeTools` before the executor's atomic state commit. `system.load_integration` is retired; integrations remain auth/workflow filters, never the model-visible activation unit.

## Authoring shapes

**Brief-only workflow.** `steps = null`, `brief != null`. Runtime: a single `AlfredAgent`-driven loop with a stable boss system prompt, the workflow brief as the first user transcript message, and exact tools seeded/preloaded/grown within `allowedIntegrations`. m12 stored and dispatched these but did not execute them; m13 added the sentinel workflow + tool dispatch + `AlfredAgent`→runtime bridge, and #411/#412 replaced integration loading with exact search/load.

**Explicit-DAG workflow.** `steps != null`. Runtime executes deterministically; node kinds: `run_skill | tool_call | llm_call | agent_run | condition | parallel | loop | hil_approve`. The Zod schema exists; runtime support for these node kinds is partial (only `agent_run` is implicit via the executor; the rest are unbuilt).

**Hybrid workflow.** Explicit DAG with embedded `agent_run(brief)` nodes — deterministic outer, LLM-decided inner. Forward-compatible; not v1.

## Privacy posture

**Content-at-rest posture.** No app-layer encryption on user content. Three layers stand in: (1) vendor at-rest crypto on Railway managed Postgres/Redis/object storage, (2) log redaction via Pino + Sentry `beforeSend` for known sensitive paths (`*.content`, `*.extracted_text`, `*.body`, `memory_facts.value`, `attachment_pages.*`), (3) `documents.raw` is not persisted — re-extraction means re-fetch from the provider. Revisit when a real second-hand party touches the data (contractor, analytics pipeline, real backup-export workflow, compliance regime). ADR-0038.

**Embedding-inversion gap.** `chunks.embedding` is plaintext by design — encrypting kills pgvector indexing. Published inversion attacks can reconstruct text from vectors + the embedding model alone. Documented and accepted; the only defense is "trust your embedding provider," which is contractual. ADR-0038.

**`SENSITIVE_LOG_PATHS`.** Single const in `@alfred/contracts` listing redacted field paths. Pino, Sentry, and any future logger pull from this one source so the redaction set never drifts across log surfaces.

## Attachments

**Attachment.** A non-text payload arriving alongside a document — PDF, image, spreadsheet. Modeled as its own row in `attachments` (sibling to `documents`, not a row in it), linked back via `parent_document_id`. Carries identity (mime type, filename, size, page count), extraction status, and binary location. Drives a distinct ingestion path because its content shape (typed segments, pages, figures) doesn't fit `documents`'s "flat text body" assumption.

**Attachment page.** One row per page of an attachment in `attachment_pages`. The citation anchor — when the agent surfaces "from receipt.pdf, page 4," it's referencing one of these. Holds the page's concatenated `extracted_text` (for "summarize this page" without rejoining chunks) and an `asset_inventory` jsonb. Page = 1 for standalone image attachments.

**Typed segment / segment `kind`.** Each chunk derived from an attachment carries a `kind`: `text | table | figure_caption | heading | footnote | list`. Drives chunking rules (tables never split mid-row; headings become standalone chunks AND populate `parent_section` on subsequent siblings) and powers retrieval queries that want a specific element type ("find the table on page 3").

**Asset inventory.** Per-page jsonb on `attachment_pages`: `{ tableCount, figureCount, footnoteCount, hasImages, hasLinks, headings: string[] }`. Counted at extraction time. Lets retrieval answer "what attachments have charts" or "which page has the financial breakdown" without re-reading every chunk.

**Doc extraction.** The pipeline that turns a binary attachment (PDF or image) into typed segments + page inventory. Implemented as a Claude call with `document`/`image` content type, prompt-shaped to emit the canonical segment JSON. Metered as `attribution.kind = 'doc_extraction'` so cost rollups bucket it apart from LLM turns / web search / embeddings. Per ADR-0021's vendor alignment and the cost calculus at single-user scale. ADR-0039.

**`doc-extraction-runs` queue.** Dedicated BullMQ queue for attachment extraction, isolated from `ingestion-runs` so Anthropic latency/cost doesn't bleed into Gmail polling. Jobs: `attachment.extract { attachmentId }` (primary) + `doc-extraction.sweep` (hourly repeatable for stuck/failed rows). Worker concurrency 5, 3 attempts, 30s exponential backoff. Terminal failure flips `attachments.extraction_status='failed'` with `last_error` populated — the row IS the dead letter. ADR-0039.

**Attachment cost gates.** Four gates bound extraction spend: (1) MIME allowlist at enqueue (`pdf|png|jpeg|heic|webp`), (2) 20MB size cap at enqueue, (3) 50-page truncation at extraction time (`truncated_at_page` flagged), (4) $5/day per-user soft cap via env `ALFRED_DOC_EXTRACTION_DAILY_BUDGET_USD` — over-budget jobs `moveToDelayed(nextMidnight())`, not lost. Skipped attachments create a row with `extraction_status='skipped'` + `skipped_reason` so the UI can show them honestly. ADR-0039.

**Attachment binary lifecycle.** Binaries land in the Railway bucket on ingest and stay forever — independent of Gmail message retention. Gmail deletion does not cascade to the bucket or the chunks. A future explicit "forget this" UI handles real deletion intent. Inverse of ADR-0038's `documents.raw` posture: there Gmail is the durable copy; here the bucket is, because Gmail attachment availability is bounded by message lifetime. ADR-0039.

## Triage

**`SenderContext`.** Output of the deterministic `extract-sender-context` step that prefixes the triage pipeline. Shape: `{ fromKind: 'person' | 'service' | 'unknown', bodyActor?: { kind, name, handle? }, effectiveAuthor: 'bot' | 'person' | 'service' | 'unknown', botSlug?: BotSlug }`. Consumed by the cheap classifier as a typed input (not raw header text) and by the briefing's email contribution. Lives in `@alfred/contracts`. ADR-0042.

**Effective author.** Derived field on `SenderContext` — the actor who actually "wrote" the email considering both `From:` and body-actor extraction. Drives classifier rule 9a/9b/9c branching and gate condition (iii) on the escalation step. `'service'` is for cases where the envelope is a service (`noreply@github.com`) but no body-actor parser fired.

**Body actor parsing.** Per-service parsers that extract the underlying actor from a service-envelope email body. v1 covers GitHub (`**actor**` markdown), Google Calendar (iCal `ORGANIZER`), and Linear ("Comment from {actor}"). Long-tail sources fall through to `effectiveAuthor: 'unknown'` and the escalation gate's `confidence < 0.7` clause acts as the safety net. New parsers added from observed `triage.sender_extraction` log evidence, not speculation.

**Bot allowlist.** Hardcoded const `BOT_SLUGS` in `@alfred/contracts/triage`. Starts at ~10 entries (coderabbit, copilot-review, github-actions, dependabot, renovate, vercel, sentry, stripe-billing, google-security, datadog). Grows from observed log data. DB-backed migration only when the list passes ~20 entries. ADR-0042.

**`SEVERITY_SUSPECT_BOTS`.** Const subset of `BOT_SLUGS` marking bots whose alerts _can_ be genuinely urgent — sentry, stripe-billing, google-security, vercel, datadog. Triage escalation gate fires on this set even when the cheap classifier output `fyi`. CodeRabbit / Copilot / Dependabot / Renovate are deliberately NOT in this set — their messages are advisory in 99% of cases and security-CVE exceptions are caught by classifier rule 9a's text-content escape.

**Deepen step.** Boss-agent brief-only run inside the triage workflow, gated by the escalation conditions. Read-only tool surface: `system.read_user_context`, `github.list_repos`, `gmail.thread_history`. This is the only v2 path where bio/profile context affects tagging; the cheap classifier remains email-only apart from `SenderContext`. Initial rollout executes only the severity-suspect-bot branch live; low-confidence and unknown-human eligibility are shadow/log-only until real inbox rates are known. Refines classifier output and optionally emits a `dossierRequest` side-effect; triage never blocks on it — failure falls back to the cheap classifier's output. ADR-0042. **Superseded for triage by ADR-0051** — triage v3 has no routine boss step; the cheap classifier is made context-rich instead (see _Observation/inconsistency layer_). The `dossierRequest` hook and `system.read_user_context` surface survive.

**Dossier cache TTL.** Per-`identity_confidence`-tier expiry on the `person_profiles` row (ADR-0031) that backs cached dossiers. ≥0.9 → 90d, 0.7-0.9 → 30d, <0.7 → 7d. Stale-and-important triggers re-research; cache key is the stable sender identifier — `email` for direct senders, `service:handle` for body actors (`github:coderabbitai`). ADR-0031 amendment + ADR-0042.

**Sender prior.** Row in `sender_priors` keyed `(user_id, sender_key)` — `sender_key` is the exact lowercased sender address or `service:<botSlug>`. Stores a raw category **histogram** (`category_counts`, e.g. `{ newsletter: 12, marketing: 1 }`), not a verdict — it is a **fed signal** to the always-on cheap classifier, never a model bypass. Updated on every classification (so it can't go stale). **Human senders are not cached** (a person's category is per-message); priors are a **bulk-sender** signal. No `confidence`/`locked`/`source`/domain in v1. ADR-0051.

**Account persona.** Per-credential `work | personal` label on `integration_credentials`, auto-detected from the Google **`hd` (hosted-domain)** claim (Workspace → `work`, absent → `personal`), user-overridable on the integration detail page. Fed as a one-line context hint to the triage classifier. v1 is **label + plumbing only**; the rich persona _policy_ (what's work-urgent vs personal-urgent) is a deferred future ADR. ADR-0051.

**Observation/inconsistency layer.** The deterministic "make a cheap model smart" mechanism in triage v3. **(a) Pre-model observations** always fed into the single cheap call (sender prior histogram, account persona, thread state, known-contact flag, Gmail-native signals, cheap regex content flags). **(b) Conditional second cheap pass** — re-runs flash-lite _only_ on a hard deterministic conflict between the model's output and a strong expectation, with the inconsistency spelled out, **max once** (the second output is final). No boss. Intelligence via _focused attention_, not a bigger model. ADR-0051. **Phase 3 seed (2026-06-05).** The conflict checker (`detectConflict`, pure) fires on two tightly-gated nets: **under-classification** (`hasSecurityKeyword` && category ∈ {fyi, done, newsletter, marketing} && override floor didn't fire) and **over-classification** (category ∈ {urgent, action_needed} && the sender prior is strong-bulk — total ≥ 5 and bulk-share ≥ 0.8 — && no security flag && not Gmail `IMPORTANT`). Gmail-signal-vs-output and the anomaly-catch direction (important output from a bulk-prior sender) are deliberately _not_ seeded — the always-on model with observations fed in already covers those. Tune from `triage.sender_extraction` logs.

**Content flags (triage).** The cheap regex `ContentFlags` fed as named observations to the classifier (`observations.ts`): `hasUnsubscribe`, `hasCurrencyAmount`, `hasSecurityKeyword`, `hasCalendarInvite`, plus (Phase 3) `hasInvestorNotice` and `hasPublicEventLanguage` — the latter two migrated from the deleted `applyTriageClassificationGuardrails` rewrites. Flags are **fed to the model and the conflict checker; they never rewrite the category** (ADR-0051 §5 — no hidden deterministic rewrites). The broad `hasSecurityKeyword` (sign-in/password/token vocab) is distinct from and broader than the override-floor predicate.

**Known-contact flag.** Pre-model observation: is the human sender's lowercased address in the user's `entities.aliases`? Computed by `isKnownContact(userId, address)` (one indexed jsonb-containment query). Only run for `effectiveAuthor: 'person'` senders (bots/services use priors instead); best-effort — any error or empty `entities` yields `false`, never fails classify. Phase 3 (2026-06-05). ADR-0051 §4.

**Triage override floor.** A small, high-precision set of unambiguous severity signals that may **force** a category regardless of model output; the model owns the category everywhere else. Seeded minimal, grows only from observed-data evidence — deliberately not a large keyword ruleset (avoids the brittleness that mis-tagged self-initiated sign-in links as `urgent`). **Phase 3 seed (2026-06-05): ONE signal** — a secret/API key/token/private key/password exposed, leaked, committed, compromised, found, or detected (in either noun→verb or verb→noun order) forces `urgent`. Generic `credential` is deliberately excluded from the unrecoverable floor and remains only in the broader security content flag. CVE-id presence and payment-failure-breaks-access are _not_ in the floor — they go to the model (a bare CVE id forcing `urgent` would re-introduce Dependabot/advisory-bot noise that rule 12a tags `fyi`). The floor predicate is narrower than the `hasSecurityKeyword` content flag so self-initiated sign-in/magic links never trip it. ADR-0051 §5.

**Thread state (triage).** Sent-mail-aware observation fed to the classifier — e.g. "you last replied in this thread on `<date>`". Requires `in:sent` ingestion. It is a **fed observation, not a hard rule**: the model owns the resulting category rather than a deterministic "you replied → done/fyi" mapping. ADR-0051.

**Sender relationship.** The directional triage observation that fixes failure A (cold inbound minting a "reply to X" todo). A triage-local resolver folded into `assembleObservations` that renders the live graph signals about the human sender as **one observation line of prose — not a typed value anything branches on**: the **precomputed** significance scalar (read at query time, bucketed to words — `strong ≥0.66 / moderate ≥0.33 / weak >0`), the reciprocity shape (one-way inbound / two-way / outbound-only), the same-org-domain flag, and the user's own role from `user_facts`. Example: `Sender relationship: weak · one-way inbound (you never replied) · not same-org · you: "Founder, Acme"`. No `relationType` enum, no self-entity, `theirDesignation` deferred to P4b. A sender with no graph row renders `no prior contact on record` → the rubric degrades to today's intrinsic-only behavior. Consumed only by rubric **16b**, and only to corroborate the *"a real person is waiting on you"* stake — money/deadline/access/commitment stakes stay intrinsic and ungated. The 16a founder/CEO/CTO title carve-out is deleted (title-proxy → real significance). Live capture folds into the daily `memory-extraction` workflow (header-only, incremental), gated by its own `memory_extraction_status.captured_into_graph_at` marker and committed atomically with the per-contact increments (at-least-once, never double-counted); people are upserted by email alias (`upsertPersonByAlias`), not display name. ADR-0059 + amendments 2026-06-16 (a, b).

## Briefing

**Briefing.** One row in `briefings` per `(user_id, briefing_date)`. The canonical record of the day's morning briefing — the email is a render of `breaking_summary`, the in-app view renders `full_briefing`. Composed once per day via a single boss-tier LLM call over a cross-source gather. Idempotent on `(user_id, briefing_date)`; status-machine lifecycle. ADR-0041.

**Breaking summary.** The 4-6 lines of markdown prose sent in the day's morning email. Generated by the briefing composer alongside the full briefing — one LLM call emits both, no drift. Resolves `[[<kind>:<id>]]` placeholders into bold + service icon + anchor per the email HTML renderer.

**Full briefing.** The longer in-app surface for the day: `{ headline, sections: { source, label, body, why?, references? }[], sourcePanels?, auditSummary? }`. Replicache-synced read-only with a 30-day pull window; older briefings reachable via an on-demand `/api/briefings/history` route. Same prose source as the breaking summary — one composer truth, two renderers. The user-facing page shows source-backed inclusion rationale, not raw model reasoning. `sourcePanels` is the normalized display model; raw `gather` remains stored for audit/replay.

**Briefing reference.** `[[<kind>:<id>]]` placeholder in composed prose. Composer emits opaque tokens; a per-surface resolver expands against the gather (email: bold + icon + anchor; in-app: typed `<EntityChip>`; plain-text: fallback label). Prevents URL hallucination and decouples LLM content from presentation. Kinds at v1: `activity | meeting | email` (closed enum in `@alfred/contracts`). Provider details such as PRs, commits, deployments, incidents, billing notices, and security alerts live inside `activity` references. Unresolved placeholders fall back to inner label as text + log to observability — drift between gather and composer is a real risk worth surfacing. **Resolution home (ADR-0049):** the pure resolver (`resolveBriefingReferences`, `BriefingSegment`, `parseBriefingReference`, `listBriefingReferenceOptions`, `referencesFromSections`) lives in `@alfred/contracts` so the web surface can resolve synced prose against the synced `gather` without importing `@alfred/api` (web-boundary safe); email's HTML renderer stays server-side but calls the same shared resolver. One resolution truth, two renderers.

**In-app briefing surface (ADR-0049).** The canonical scrollable record of a day's briefing at `/briefings` (timeline) + `/briefings/$date` (day detail). Day-keyed: the detail prefix-scans `briefing/{date}/` via `IDB_KEY.BRIEFING({ id: \`${date}/\` })`and renders **morning + evening stacked** as two slot records. Render posture is **paragraph-first** per ADR-0048 — lead with`breaking_summary`, treat `sourcePanels`as scrollable supporting detail and`sections[]`as secondary/collapsible, never the primary spine. **No special quiet-day mode:** every row renders identically; a quiet morning simply composes to a shorter`full_briefing`, so it *looks* quiet without special chrome. `suppressed` (morning not emailed) is at most a subtle "Not emailed" metadata line, not a distinct layout. Read-only via Replicache (30-day window); no client mutators.

**Gather source.** A contribution to the briefing's input. Each source registers a `BriefingContributor<T>` exposing `collect({ userId, date, timezone }) → T | null`. v1 sources: `email | calendar | integration_activity | weather | day_of_week`. Future connected providers add activity producers under `integration_activity`; they do not each become top-level briefing sources unless they need a distinct briefing role. Provider names like Vercel, Railway, Cloudflare, Sentry, or Notion are examples, not committed integrations.

**Integration activity.** Normalized operational updates across connected systems, with GitHub as the first planned direct producer and other providers added only when their integrations exist. Items carry a stable `activityCategory` (`work | deploy | incident | account | billing | security | usage | other`) plus provider-scoped `providerKind` strings (for example `github.pr_review_requested`, `some_provider.deployment.failed`). Raw events are rolled up before compose: transient failures followed by success are suppressed unless the cluster is notable (many attempts, long duration, critical severity, or clear user involvement). Email triage can backfill activity with `source='email_triage'` before a direct provider API exists.

**`IanaTimezone`.** Branded `string` type in `@alfred/contracts`, validated at the API boundary against `Intl.supportedValuesOf('timeZone')` (the live IANA list, ~400 zones). Persisted as `text` (no PG enum — IANA mutates per tzdata release). Used wherever the user's timezone has to round-trip through DB + API + Replicache safely, including `briefings.timezone`.

## Todos

**Todo.** One row in `todos`. A user-managed commitment — a checkbox item either captured by the user or proposed by Alfred. The **first persisted materialization of the open-loop model** (ADR-0048 keeps loops ephemeral at briefing compose-time; a todo is a loop the user has chosen to track). Single table, status-driven; Replicache-synced. v1 is **passive** — Alfred authors and assists but never executes (see _Agent-executable todo_). ADR-0050.

**Todo-worthiness rubric.** How Alfred decides an email warrants a `suggested` todo (ADR-0050 amendment 2026-06-06) — a judgment **orthogonal to the triage category**, applied by the cheap classifier over the whole email (including secondary/trailing asks) as five ordered tests, stopping at the first failure: **(a) obligation on me** (gate — an action the _user_ must take, AND the named owner is the user, not a third party named in the body — the Sakshi-standup attribution fix, 2026-06-09; needs a minimal identity signal: user display name + account email), **(b) real, external stake** (significance reframed 2026-06-09 — a real identifiable person waiting on the user, money owed/at-risk, a hard deadline, lost access, a commitment to a human, OR a real-world consequence to the user judged from content (this is where a _bot_ earns a todo — leaked key, prod down); **manufactured/ceremonial urgency** — engagement streaks, unread counts, "X viewed your profile", marketing scarcity, AGM/"save-the-date" — carries no stake and fails here; intrinsic-only in v1, personal-relevance still deferred → ADR-0050 D1, but written as a B-ready seam), **(c) memorability** (would the user forget/drop it, or is it self-resolving / mid-flow — this is what kills self-initiated auth, as a _consequence_ not a patch), **(d) actionability** (a specific item is writable from the email alone), **(e) already-handled** (thread state shows it's done). Only all-five-pass yields a `todoSuggestion`. **Category and todo may disagree** (a `done` closure with a real trailing ask → todo; the category is never bent to fit). The deterministic floor `TODO_INELIGIBLE_CATEGORIES = {marketing, newsletter}` is a _consistency guard against classifier leakage_, NOT part of the rubric — `fyi`/`done` go through the rubric. The decision is traced on every classification via **`todoDecision`** `{ outcome, note? }` (`outcome ∈ {proposed, no_obligation, not_significant, would_not_forget, too_vague, already_handled}`), logged to `triage.sender_extraction` so the rubric is **tuned from real misses, not example-patched**. A small set of **exemplars** in the prompt anchor the rubric only at dimension boundaries, harvested from logs. Prompt rule 16 + `resolveTodoSuggestion` in `classify.ts`. **Amendment 2026-06-09:** the real-stake/manufactured-urgency principle ALSO reinforces the _category_ side (ceremonial "meetings" like AGMs → fyi, generalizing rules 8/9); the **bot carve-out is model-judged from subject/content, NOT a hardcoded `BOT_SLUGS` floor** (advisory automated output → fyi/no-todo unless the body carries a hard consequence); titles must be terse imperative active voice. Validated by a **dry-run backfill** that re-classifies historical email and diffs categories + suggestions before any writes.

**Todo status.** `suggested | open | done | dismissed`. `suggested` = Alfred-proposed, not yet accepted (renders in the _Suggestions_ section). `open` = a live todo (user-added, or a promoted suggestion). `done` = completed (`completed_at` set; lingers in sync for 7 days, then only falls out of the pull window). `dismissed` = declined/dropped — on a `suggested` row it's a "no thanks" learning signal, on an `open` row it's "dropped it." Promote (`+`) flips `suggested → open`. There is **no `running`/`interrupted`/`needs_attention`/`error`** at v1 — those are dimension's agent-run states, reserved for the deferred agent-executable path.

**Suggestion (todo).** A `todos` row with `status='suggested'`, **not a separate entity** (unlike dimension's `suggestedMessages`). Rendered in the rail's _Suggestions_ section with a `+` to promote. Authored only by Alfred (`created_by='agent'`). Carries `assist` + `sources` like any todo.

**`created_by`.** `'user' | 'agent'` on a todo. Survives promotion (a promoted suggestion stays `created_by='agent'`), so suggestion acceptance/dismissal is measurable later.

**Todo `executor` / `kind`.** Forward-compat columns. v1 rows default to `executor='user'` and `kind='task'`; the columns are inert until the deferred agent-executable path needs `executor='agent'` and executor-specific variants.

**`assist`.** Optional Alfred-authored text on a todo — a tip on how to approach it, degrading to an honest "I can't act on this (no permission / integration not connected)" when Alfred is clueless. Present even in passive v1; it is _not_ execution.

**Todo `sources`.** `jsonb` array of typed cross-source provenance refs — `[{ provider, kind, id, url? }]` (e.g. a Gmail thread _and_ a Slack thread on one row). Canonical identity is `(provider, kind, id)`; `url` is display/navigation metadata. Multi-source from day one so a todo represents a real-world commitment, not one channel. `created_by`, `agent_run_id`, and `assist` round it out. Single email-only `source_thread_id` was rejected for this reason.

**`system.suggest_todo`.** The source-agnostic write tool by which any agent run proposes a todo (inserts a `suggested` row). **No HIL/approval** — a suggestion has no real-world side effect, so it stays off the `action_stagings` path; audit lives on the todo row (`agent_run_id`, `created_by`, lifecycle dates). **Produced real-time off the `email-triage` run** — a tail step calls it when the cheap classifier emits a `todoSuggestion`, governed by the **Todo-worthiness rubric** (ADR-0050 amendments 2026-06-05 / 2026-06-06); the briefing does **not** call it (its action mentions are ephemeral open loops, ADR-0048). Source-agnostic by design, so future `integration_activity` producers call it from their own ingestion paths. **Idempotent on source-ref overlap**: if an existing `open`/`suggested` todo already references an incoming source, it **merges** (appends only missing refs) rather than creating a duplicate — this is the v1 cross-channel dedup guard.

**Agent-executable todo (deferred).** The B-path: a todo marked `executor='agent'` that spawns an `agent_runs` run through the boss runtime, with run-state (`running`/`interrupted`/`needs_attention`/`error`) reflected on the checkbox and tool calls gated via the existing `/approvals` HIL. Dimension ships this; Alfred defers it but shapes the schema (forward-compat `executor`/`kind` columns) so it lands without a database migration. The screenshot's checkmark/camera toggle is this switch.

**Cross-source auto-close.** Closing a todo automatically when integration activity proves the underlying task is done. ADR-0048's "continuous open-loops state machine" — **GitHub design locked (ADR-0052), read-only PR search shipped; poller/reconciler columns deferred**. Future v1 GitHub: a poller produces + reconciles persisted todos from the authenticated API; a _positive_ terminal signal (PR merged, alert dismissed, review threads resolved) flips `suggested→dismissed` / `open→done`, **absence never closes**. Generalizing to every integration (Slack reply closes an email-born todo) still needs cross-surface identity resolution and is deferred; the `sources` array is the watch-list each todo already records. Manual check-the-box remains the universal fallback.

**GitHub loop poller (deferred).** The ADR-0052 job (~10-min BullMQ tick). **Two-tier:** `GET /notifications` (the structured mirror of GitHub's notification emails — PRs, issues, Dependabot alerts, CI, mentions) detects + produces; per-object state endpoints give the authoritative close signal. A **deterministic `reason` gate** (no LLM) decides worthiness; a **cheap LLM** authors the title only for gate-survivors. Produces `suggested`. Email-derived identity is explicitly _not_ used.

**Verified-sender suppression (deferred).** The cross-surface dedup guard (ADR-0052): a DKIM/SPF/DMARC-verified `github.com` notification, when GitHub is connected, makes email triage **skip** minting a todo (the poller owns that loop with native identity). Fail-safe toward surfacing — unverified mail falls through to normal triage. Net-new `Authentication-Results` capture in the Gmail ingest path.

**Todo closure signal (deferred).** `todos.closed_by` (`'user' | 'reconciler'`) + `closed_reason` (`pr_merged | pr_closed | issue_closed | alert_dismissed | alert_fixed | review_threads_resolved`), `.$type<>()`-guarded. One field, three jobs: audit, the evening-recap "loops closed" line (ADR-0048 #5), and the client's auto-dismiss animation (reconciler-closed rows linger briefly in the Replicache pull so the diff is visible).

## Meeting prep

**Meeting prep packet.** Alfred's pre-meeting context note for one calendar event: who's attending, the recent threads with them, relevant memory facts, and open todos that touch the meeting — composed into a short cited note. **MEET-001 scope (locked 2026-06-10/11):** mirrors the briefing pipeline end to end — a deterministic **gather** (event + attendees + email threads + memory facts + todos), a **boss-tier compose** into a short prose note with `[[<kind>:<id>]]` citations, **persisted** to `meeting_preps` and Replicache-synced read-only. The proactive **calendar-watch trigger** is folded into MEET-001; the **delivery surface** (pre-meeting email, web augmentation at send) is MEET-002. See [docs/plans/meeting-prep-v1.md](./docs/plans/meeting-prep-v1.md), proposed ADR-0054.

**`meeting_preps`.** One row per calendar occurrence, keyed `(user_id, event_key)` where `event_key = ${credentialId}:${googleEventId}` (`singleEvents=true` → per-occurrence). Status machine (`pending|gathering|composing|ready|failed|cancelled`) like `briefings`; `gather` jsonb is the audit/replay record, `note` jsonb the composed cited output, `row_version` drives Replicache. **Upsert, recompute-in-place** — no version history; cancelled calendar occurrences are marked `cancelled` and fall out of active sync.

**Gated recompute.** The deterministic (no-LLM) discriminator on re-running `prepareMeeting`: a `material_hash` over attendees + agenda + location + attachments. No row → full gather+compose; material change → full recompute; **time-only shift → cheap path** (update `event_start`, skip compose); unchanged → no-op. The lever that lets the trigger fire often while keeping boss-tier compose cost down — minimize the *number* of composes, not the tier.

**Prep horizon + sweep.** Prep *compose* fires only for qualifying events starting within horizon **H (48h)**. A calendar push preps near-term changed events; a ~20-min cron **sweep** catches events crossing into H that never changed (the webhook-vs-time-passing gap — pushes fire on change, not on time). Event qualification on window-resolution: timed, `attendees ≥ 2`, not declined; an explicit `eventKey` bypasses qualification.

**Calendar watch.** Google Calendar push channel (`events.watch`) — **HTTPS-callback** based, not Pub/Sub like Gmail. State lives in `integration_credentials.metadata.calendarWatch` `{ channelId, resourceId, expiresAt, syncToken, calendarId }` (mirrors Gmail's `metadata.watch`; v1 = `primary` only). **No event mirror** — push handler acts on the `events.list(syncToken)` delta live, sweep/gather read live. Push delta dispatches through the `emitEvent` bus (`calendar.event_scheduled`, ADR-0047). Needs a domain-verified callback URL (GCP) — no localhost in dev.

**`system.prepare_meeting`.** The single entry point all three triggers (chat/boss, calendar push, sweep) converge on. `system.*` → autonomy by default, riskTier `no_risk`. Input `{ eventKey } | { timeMin, timeMax, attendeeHint? }`; resolves the event, runs the gated recompute, gathers, composes, upserts `meeting_preps`.

**Prep reference.** `[[<kind>:<id>]]` placeholder in composed prep prose. Parallel to the *Briefing reference* but its own closed enum + resolver in `@alfred/contracts` (`MEETING_PREP_REFERENCE_KINDS = [meeting, email, todo]`, `resolveMeetingPrepReferences`), expanding against the prep gather — briefing's resolver is untouched. `meeting:<eventKey>` static, `email:<documentId>` → Gmail thread, `todo:<todoId>` → rail. **Memory facts are not a citation kind** in v1 (woven into prose; ids retained in gather for a future SEARCH-001 evidence layer).

## Long-term memory

**Memory store posture.** The substrate is **Postgres, not a graph DB** (ADR-0058). Two separable layers: the **store** (where memory lives — stays PG: `user_facts`, `entities`/`entity_relations`, `memory_chunks` pgvector) and the **intelligence** (extraction, contradiction, lifecycle — *built* as logic over the store, not bought). Graph is kept as a **model** (`entity_relations`), never a graph **engine** — the engine's advantage is a node-scale single-user Alfred never hits, and a service (Zep/Mem0/Letta) drags the source of truth off Postgres, breaking the Replicache memory-changelog surface + the ADR-0038 privacy posture. Borrowed: Mem0's `ADD/UPDATE/DELETE/NOOP` taxonomy as the *shape* of write-time contradiction over `proposeFact`/`supersedeFact`. Revisit only on multi-tenant-at-scale.

**Memory governance.** ADR-0056 posture: Alfred mutates memory **autonomously via `system.*` tool calls**, kept safe by **transparency + reversibility**, not pre-write HIL. Supersedes ADR-0019's confidence-gated accept gate — confidence now gates _notification cadence_ + the `proposed`/`confirmed` _review label_, not the write. Floor unchanged (≥0.7 written, <0.7 dropped); the agent **hedges on low-confidence** facts in prose. Storage substrate adopted as-is.

**Knowledge kind.** The taxonomy spine memory is organized by (not by table): _identity_, _standing instructions_ (first-class), _people & relationships_ (the team graph), _episodic facts_, _style_, _episodic memory_. Each has its own capture source + lifecycle; **lifecycle is per-kind — there is no global TTL.** ADR-0056.

**Criticality tier.** Drives notification, not gating. _Critical_ (identity change, key-relationship change, superseding a ≥0.85 fact, retracting a confirmed fact) → email on a **~5-min debounce, batched** (reuses the approval-debounce). _Subtle_ (new low-stakes/low-confidence facts, additive aliases) → **digest flushed on count-threshold OR weekly, whichever first**. In-app is real-time Replicache, **one-by-one** (`user_facts.row_version`). ADR-0056.

**Rejection cause.** New field on the rejection record: `cause ∈ {user, write_time_contradiction, decay, superseded_by_newer}` alongside the freeform `reason`. Separates _user corrections_ (high-value training signal) from _system self-corrections_. ADR-0056.

**Memory rationale.** A cheap-model **~2-sentence, telegraphic, grammar-optional, token-frugal** "why" persisted per memory write, paired with the `source` evidence pointers. Surfaced through the **SEARCH-001** evidence layer — memory-justification and cited outputs are one mechanism. ADR-0056.

**Loop 1 / Loop 2.** Two self-correction loops a single user correction drives at once. **Loop 1** = fact maintenance (write-time-contradiction supersession + user-authoritative edit/reject; decay deferred → D2). **Loop 2** = system learning (the user-correction misses dataset, seeded by `rejected_inferences`, feeds the eval lane ADR-0055; **no automatic prompt mutation** — humans gate). ADR-0056.

**Passive capture.** Memory capture is **fully passive** — onboarding stays minimal; the team graph + identity facts are inferred from integrations (Gmail/Calendar) and **enriched/corroborated via web search**, written autonomously (ADR-0056), kept fresh by continuous extraction + Loop-1 supersession. No prompted confirmation card; correction is unprompted via the review surface. Teams change → the change shows up in an integration, so no re-interrogation. **Supersedes ADR-0031's review-before-write gate** for dossiers; `person_profiles` (ADR-0042, unbuilt) is the dossier home. ADR-0057.

**Significance score.** A first-class **computed signal over `entities`** — blend of correspondence frequency + recency + reply-reciprocity + same-org-domain + explicit relationship edges. **One source, four consumers:** the web-search dossier-enrichment gate (budget-capped, only above-threshold entities get a dossier; long tail stays name+email), todo personal-relevance significance (ADR-0050 **D1**), triage sender priority, meeting-prep attendee prioritization. Replaces four ad-hoc "who matters" heuristics with one. The score stays a **scalar**; directional richness lives in the _Sender relationship_ resolver, not here. ADR-0057.

**Sender relationship.** The **triage-local, directional** resolution of a sender — `{ relationType, direction, theirDesignation, yourRole }` — composed at classify-time from the _significance score_ (scalar) + the sender's `entities` row (org/designation) + the user's identity `user_facts` (`company`/`job_title`) + a shared-org-domain test (sender domain ∈ the user's account domains). Answers "who am I _to this sender_" so todo rubric 16b can judge **relationship-derived** stake, not just intrinsic. **Consume-not-infer:** 16b reads this block, never hallucinates a relationship beyond it; when the graph is empty it returns `stranger`/`no-data` and 16b degrades to intrinsic-only (today's behavior). **No self-entity** — direction is derived, `user_facts` stays the single source of truth for who the user is. Distinct from the significance score (one scalar, four consumers); only the todo-D1 consumer needs the edge, so it lives in triage. ADR-0059.

**Chat→memory capture.** The explicit-statement write path (vs _passive capture_'s inference). Two paths: **(i) in-band proactive** — the boss recognizes durable intent mid-conversation (not just literal "remember") and calls `system.remember`; **(ii) end-of-thread extraction** (ADR-0019's existing trigger) mines the closed thread for durable facts/prefs stated in passing. Bar to write is low (reversible); behavior-changing standing instructions are critical-tier → notify. **Durable vs run-scoped split:** "for the rest of this conversation" → run-scoped directive (ADR-0035 `user_directives`, dies with the run); "from now on" → durable standing instruction (persisted). The boss classifies on phrasing; only durable hits long-term memory. ADR-0057.

**Standing instruction.** A durable, user-stated **bias on an existing pipeline** (triage / briefing / meeting-prep) — "tag my partner's mail urgent", "stop nagging me about Ben", "keep recruiters out of my brief". A `user_facts` row, `key="standing_instruction"`, consumed by the pipelines as an orthogonal modifier. **Boundary (decided 2026-06-17):** a standing instruction *modifies a run that already happens* — it is **not** a request to *author new scheduled behavior*. "Brief me before the eng standup" is workflow-authoring (ADR-0017 skill/workflow + ADR-0027 trigger → an `agent_runs` trigger), **not** a standing instruction; the two may share the chat capture entry point (the boss decides which artifact to mint) but are different artifacts with different lifecycles. Keeps this from drifting into a generic rules engine. The _Suppression standing instruction_ is the first concrete shape.

**Standing instruction schema (v2).** `user_facts.value`, `schemaVersion:2` — three cleanly separated concepts (2026-06-17): **`directive`** (resolved prose, **always present** — the universal carrier prose-consumers apply by judgment), **`target?`** (optional matchable descriptor — `{kind: sender_email|sender_domain|person|category|topic, …resolved-at-write key}` — serves both deterministic exact-match and prose relevance-filtering; absent for a pure free-text rule), **`enforcement?`** (optional typed+parameterized actions for the deterministic carve-out — `block/force_todo_suggestion`, `force_category{category}`, `exclude/include_briefing_priority`; **registered, not closed**), plus `phrasing` (verbatim, provenance/UI only). **`action`/`surface` are dropped** (v1 → v2; `directive` carries display intent, `enforcement` carries the contract; reader handles v1, near-free migration). The shipped v1 _Suppression standing instruction_ is the `enforcement`-bearing special case.

**Instruction application model.** *How* a standing instruction reaches a pipeline — **prose-first, with a deterministic-enforcement escape hatch** (redirected 2026-06-17, supersedes the earlier per-consumer "effects enum is the contract" framing). The architecture has three parts, central by design so a new feature/integration costs ~nothing:
- **Central store + one retrieval primitive.** All instructions live in `user_facts` (`key="standing_instruction"`). A single `getRelevantInstructions(context)` retriever returns the instructions relevant to whatever a feature is doing (an email, a briefing gather, a meeting, a chat turn). A new consumer *calls it* — it never extends an enum. **Relevance strategy = inject-all-active** (the set is bounded) **behind a context-aware interface** — semantic/entity relevance-filtering is a deferred swap behind the same signature, triggered by a logged count threshold, never a call-site change. **No-double-application:** an instruction with an `enforcement` is applied by the deterministic consumer that owns that effect; prose-fallback fires only for instructions a consumer hasn't already enforced (so "suppress Ben" is dropped by `gather` and never reaches `compose`).
- **Prose-first application (the default).** The resolved **`directive`** prose is the universal carrier. Agent-driven, compose-time, low-volume consumers (chat, briefing compose, meeting-prep, **every future agent feature**) pull the relevant directives and **apply with judgment** — including flagging conflicts / counterproductive effects ("you also told me to always surface Ben"; "muting all of @oliv.ai hides your manager"). No typed effects, no per-feature maintenance, instruction-awareness for free.
- **Deterministic enforcement (the carve-out).** A thin typed projection (`enforcement`: e.g. `block_todo_suggestion`, `force_category{category}`, `exclude_briefing_priority`) consumed only where a **hard guarantee + cost-sensitivity coincide** and a prompt line is not reliable enough — **triage today** (the cheap, high-volume classifier; ADR-0051 keeps it context-free, the Ben-Book veto proved prose-only is insufficient there). **Keyed on "needs a guarantee," NOT on feature count** — it's a registry, so a *future* feature that needs a hard guarantee registers an enforcement, but ordinary new features just use prose-pull. The shipped slice's `effects` enum is this layer, scoped down.

The boss picks at capture (resolve-at-write): prose-only `directive`, or `directive` **+** a resolved `target` + `enforcement` when a deterministic guarantee is needed. Likely an **ADR** (architectural, hard to reverse).

**Category override (user-instructed).** Refines the "category stays honest" rule (2026-06-17): **the *model* never bends the category; an explicit *user* standing instruction may** — via the deterministic override layer, never by the classifier. Carried by a parameterized **`force_category`** effect (`{effect:"force_category", params:{category:<TriageCategory>}}`). The model still classifies honestly; the override layer re-stamps *after*, exactly like the existing security override-floor in `classify.ts`. **Every override logs an audit breadcrumb** (`overriddenByInstruction: { factId, from, to }`) — non-negotiable: a future debugger must never see a force-stamped category with no trace. Distinguished from the model's own "no bend" rule because the user is the authority, not a hallucination. Reversible (reject the fact → override stops).

**Suppression standing instruction.** The first concrete standing-instruction shape (the "stop emailing me about Ben Book" slice). A `user_facts` row, `key="standing_instruction"`, `value = { schemaVersion:1, action:"suppress", surface:"open_loop", target:{ kind:"sender_email", email:<canonical lowercased email>, label:<display>, accountId:null }, effects:["block_todo_suggestion","exclude_briefing_priority"], directive:<resolved prompt-ready sentence>, phrasing:<verbatim words> }`. **`effects` is the operational contract consumers branch on; `surface` is the human/product label** ("does `open_loop` include me?" is answered by `effects`, never re-derived). **Sender-scoped, cross-account by default** (`accountId:null` — suppress the *sender*, not one mailbox); `surface:"open_loop"` = suppress the **nag/todo/briefing surfacing**, not the email's existence (the chat reply says so explicitly — emails still arrive unless the user asks for a Gmail filter). **Persist-until-revoked** (no `valid_until`; a later "surface Ben again" supersedes via `supersedes_id`). **One shared reader** (`listActiveSuppressionInstructions`/`isSenderSuppressed`) owns the zod parse + `proposed|confirmed` filter + `valid_until` filter + lowercased-email match; triage `classify` (blocks `todoSuggestion`) and briefing `gather` (excludes the sender from priority buckets) both call it — an **orthogonal check, NOT a category bend** — and each logs an audit breadcrumb (`suppressedByInstruction`/drop count). Topic-scoped suppression (fuzzy → prompt-inject) is deferred. ADR-0058 slice; [docs/plans/long-term-memory-v1.md](./docs/plans/long-term-memory-v1.md).

**Instruction precedence (apply-time).** When several non-conflicting instructions match one item (2026-06-17): **most-specific target wins, recency breaks ties** — specificity order `sender_email`/`person` > `sender_domain` > `category` > `topic`; same-specificity → newest (capture's conflict check usually catches same-target contradictions first). `enforcement` stays authoritative over the model's intrinsic judgment for the dimension it touches. **Safety carve-out:** the existing protective override-floor in `classify.ts` (force-`urgent` on exposed-secret / account-compromise) **outranks a user *down-rank*, never a user *up-rank*** — a mute can't bury an irreversible-harm alert; an elevate stacks fine. Named, floor-only, logged like every override.

**Target disambiguation (resolve-at-write).** When the user's words match **more than one** known entity ("Ben" → two people in the team graph), canonical resolution is still a write precondition: the boss **surfaces the candidates and asks which** rather than guessing — the same fragility that made the original boss search "ben book" and hit its own reminder emails. Person/sender targets resolve via the team graph (`upsertPersonByAlias`/`entities`); >1 candidate → disambiguate, 0 confident → one-line clarifying question, exactly 1 → proceed.

**Conflict at capture.** A write is gated on a conflict check, two flavors (2026-06-17). **Structural conflict** (overlapping `target` + opposing `enforcement`, or target subset/superset) is detected **in the `system.remember` tool** — returns `status:"conflict"` + the conflicting fact(s) + a machine reason (mirrors `resolve_todo`'s `ambiguous`, extends `already_exists`); **the tool never silently overwrites a contradiction** (relying on the model to notice is the prose-only fragility again). **Semantic/counterproductive conflict** ("mute all of @oliv.ai" hides the user's manager) is the **boss's** job — reasoned from existing instructions + `read_user_context` (team graph) before writing. In both cases the boss **poses a counter-question / offers options** rather than writing through; a confirmed reversal supersedes via `supersedes_id`. **Offline/passive capture** that hits a conflict **holds the write as `proposed`** (surfaces in the memory changelog) — never silent overwrite.

**Resolve-at-write.** A standing instruction is **fully resolved at capture time** so no consumer ever re-interprets prose at read time: `target.email` holds the *canonical matchable key* (the resolved, schema-normalized lowercased email, not the name "Ben Book"), `directive` is a resolved prompt-ready sentence, and `phrasing` (verbatim) is **provenance/UI only — never parsed by a pipeline**. Canonical resolution is a **precondition of the write**: if the boss can't confidently resolve the target from thread context, it asks a one-line clarifying question rather than guessing (the antidote to the original bug, where the boss searched "ben book", hit its *own* reminder emails, and mis-acted). Capture path = two boss tools: `system.remember` (writes the row, ADR-0056 governance) + `system.resolve_todo` — **source/sender-resolving, not `todoId`-only** (`{ todoId? | senderEmail? | source? }`, mirrors `suggest_todo`'s source-overlap match, returns `ambiguous` + candidates when >1), and uses **`dismissed`** (not `done` — `done` trips the 7-day done-sync/Gmail-label window) for spam/suppression. The tool whose absence let the boss falsely claim "I marked it resolved".

**Standing instructions surface.** A minimal-but-real review panel is **in scope** for the generalized ship (2026-06-17, supersedes the slice's "polished changelog deferred"): a "Standing instructions" list (likely `/settings`, near the Background card) of each active rule's plain-English `directive` + **undo/reject** + **edit**, reading the already-synced facts (`row_version`) and reusing the existing reject/edit mutations — no new sync. Required *because* positive instructions **actively mislabel** mail (a stale `force_category` distorts triage silently — the exact failure that started this), so audit can't be logs-only. **Notification unchanged (ADR-0056 + slice):** a chat-stated instruction sends **no email** (user witnessed it); only passive/background-captured instructions notify.

**Recurrence-decay (briefing).** The briefing principle that defuses the self-reinforcing nag loop: **un-acted-upon recurrence decays salience — it never escalates it.** An item surfaced across N briefings with no user action is a signal the user *de-prioritized* it → downgrade or drop, never amplify. Inverts the emergent "survived every briefing, shouldn't survive another night" escalation that `list_prior_briefings` was driving. **Generalize the existing PR-scoped "Don't re-surface stale PRs" paragraph** (`briefing/prompt.ts`) to cover any recurring item — a prompt **principle**, not an exemplar (principles-over-exemplars), shipped as an **eval-backed prompt change (ADR-0055), NOT a slice build-blocker**: the deterministic suppression check is what closes the loop; recurrence-decay is defense-in-depth for *un-suppressed* items ("a fix in triage is not a fix everywhere"). ADR-0058 slice.

**Cold-start research (v2).** Lifetime-once onboarding self-research from name + email (ADR-0011 lineage). Scope is **the account holder only**: company, role, team, and publicly attested relationship/family facts may land as `user_facts`/`memory_chunks` _about the user_; third parties get at most a one-line relation mention, never recursive research or their own `entities`/`person_profiles`. The team graph is ADR-0057's later passive, significance-gated job. The blocker at onboarding is **not "no history exists"** (a connected mailbox brings years of it) but that the history is **not-yet-ingested / not-readable under the `gated` default** — `gmail.read_message` reads the empty `documents` store at callback, and live reads need the unbuilt run-scoped autonomy override. Hence population is passive-capture's job, not cold-start's: **P4a** backfills the team graph over already-ingested correspondence; **P4b** (onboarding header-level seed) waits on the override (ADR-0059).

**Cold-start v2 shape.** Replaces stranded Perplexity Sonar Deep Research (ADR-0011/0022) with the agent harness: boss identity-resolution seed -> bounded parallel aspect sub-agents using `system.web_search` -> boss synthesis -> existing `extractColdStartFacts` -> `proposeFact`/`writeMemoryChunk`. **v2.0 is web-only** because new users default to `gated`, which would park live Gmail/Calendar reads in a watcher-less onboarding run. **v2.1** may add read-only `calendar.list_events` + `gmail.search` after the run-scoped autonomy override lands; `gmail.read_message` remains unavailable at callback time because it reads the not-yet-ingested `documents` store. Caps: sub-agent findings ~500w ephemeral; boss synthesis ~**300w telegraphic** and persisted as the `memory_chunk`/extractor input; extracted fact limits stay unchanged. v2 keeps ADR-0019's two-stage extract tail and does not pull in `system.remember` or the ADR-0056 governance migration. **Relation guard:** attestation, not fame; never infer from surname/coincidence; hedge or omit low confidence; for minor children, "exists/how many" is enough and individual background is omitted. ADR-0011/0022 amendment (pending, in design).
