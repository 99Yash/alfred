Who owns what?

Use this when “MCP,” “the agent,” and “the integration” start collapsing into one black box.

ConcernPrimary ownerMCP’s contribution
Interpret the user’s goalHost + modelNone prescribed
Choose model-visible capabilitiesHostServer discovery supplies candidates
Describe callable inputs/outputsServerTool metadata + JSON Schema
Propose a callModelNone; happens in the host’s model API
Authorize this user to the serverServer + authorization server + clientHTTP OAuth framework and audience binding
Allow this action in this workflowHost policyNone prescribed
Ask for human approvalHost UI/policyElicitation can carry server-initiated requests; it is not the host’s action gate
Validate argumentsHost and serverShared schema and error envelopes
Perform the external side effectServer / downstream APItools/call transports the request
Sanitize and bound returned contentServer and hostTyped content envelope; no trust guarantee
Retry safely after ambiguityHost + server semanticsErrors/tasks help coordination; no exactly-once guarantee
Checkpoint, resume, and audit a runHost runtimeNot MCP’s scope
Keep a huge tool catalog cheap for the modelHost + model providerDynamic listing helps; prompt/cache strategy is separate
Sandbox a local server processHost/OSSecurity guidance, not automatic enforcement

Diagnostic rule: if the question contains “should,” “may this user,” “retry,” “resume,” “trust,” or “undo,” the answer is almost never “MCP handles it.”

One-line wire flow

initialize → discover → model proposes → host authorizes → tools/call → server enforces → result → host sanitizes/checkpoints → model continues
